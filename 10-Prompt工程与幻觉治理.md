# 10 · Prompt 工程与幻觉治理 · 抖店诊断 Agent 实战

> **文档定位**：配套 `00-项目说明书.md` 使用。
>
> 面试材料里 `05-面试真题应答框架手册.md` 的 Q5 / Q6 / S8 给的是**通用模板**（5 层防幻觉、3 层防护、广告法词库）；本文档是**这些方法在抖店诊断 Agent 项目里具体怎么落地**——给面试官追问时能展开讲 5-10 分钟，不至于只能背模板。
>
> 核心论点：**本项目的防幻觉不靠 prompt 调魔法，靠架构**——Skill MD + 双层 Wiki + evidence_refs + Review 四件套形成结构性约束，prompt 只是把约束传达给 LLM 的接口。

---

## 0. 一句话

我这个项目里，**prompt 不是核心防线，架构才是**。任何把"模型自己想清楚"作为防线的设计在严肃交易场景都会翻车——抖店商家拿这个报告去做决策，每一条建议都关系到一周营收，没法用"AI 偶尔说错"的姿态搪塞。

---

## 1. 本项目独有的幻觉风险

抖店诊断 Agent 的幻觉是这 6 类——只有第 6 类是普通 LLM 输出问题，**前 5 类全是"AI 把不知道的事编得像那么回事"**：

| 类型                            | 表现                                                                                | 后果                       |
| ------------------------------- | ----------------------------------------------------------------------------------- | -------------------------- |
| **① 指标基线编造**       | AI 说"类目转化率 2.4%"，但 Wiki 里没这条                                            | 商家照编造的数字决策，全错 |
| **② 资源 URL 编造**      | AI 说"申请商城精选（→`tool://shangcheng_jingxuan_apply`）"，但抖店根本没这个入口 | 商家点不开，产品信任崩     |
| **③ 准入门槛编造**       | AI 说"618 活动准入需要体验分 ≥ 4.5"，但实际是 4.3                                  | 商家白冲一周高退款 SKU     |
| **④ 根因瞎编**（最难治） | 看到 CVR 跌就说"主播话术不行"，但实际是千川 ROI 倒挂的连锁反应                      | 商家做了无效动作           |
| **⑤ 跨 case 串台**       | 给货架店出"调福袋节奏"建议（应该是自播店建议）                                      | overlay 路由错位           |
| **⑥ 数字幻视**           | "估算多带 20w" 但底层数据不支持这个数字                                             | 商家不达预期会投诉         |

**6 类幻觉对应的治理思路完全不同**——不能用同一套 prompt 模板治所有。下面 §2 / §3 按类型对症下药。

---

## 2. 架构层防幻觉 · 四件套

在我把 prompt 写得更花之前，先看架构能挡住多少。

### 2.1 Skill MD 强约束 —— 治 ④ 根因瞎编

- 每个专家 1 份 markdown，明确写：角色 / SOP / 工具依赖 / 输入 schema / **输出规范**
- "输出规范" 段写硬约束 + 红线：例如 `agent/skills/xingdong_jianyi.md` 里写「**每条 action 必须挂 resource.url，禁止空建议**」
- **关键点**：业务侧能改 MD —— 真值放在业务侧的可读文件里，**不放在模型自己心里**
- 防住 ④：归因专家 SOP 明确列出 MECE 维度（流量 / 转化 / 商品 / 履约 / 用户 / 价格），LLM **不能自创维度**

### 2.2 双层 Wiki 事实底座 —— 治 ① 基线编造 / ③ 门槛编造

- 类目基线 / 活动准入 / 工具入口 / 规则变动 **全在 `agent/wiki/industry/*.md`**
- Attributor / Advisor 节点的 prompt 里 **强制塞入 Wiki 检索片段** 作为上下文
- LLM 出"类目转化率 2.4%" 之前，必须 Wiki 里有这条；没有就退化为"暂无类目基线，仅同比"
- 防住 ③：体验分准入门槛在 `activity_db.md` 写死，规则代码校验 `eligibility.met`，**不让 LLM 自评"够不够格"**

### 2.3 evidence_refs 强制引用 —— 治 ① / ② / ③ 全部

- 每个 anomaly / root_cause / action 必须有 `evidence_refs[]`
- 每条 evidence 是 `{refId, layer, sourceId, quoteOrSummary, confidence}`
- `layer` 只能是 `raw`（来自店铺数据）或 `wiki`（来自行业 Wiki）—— **没有第三种**
- Composer 节点过滤：**没有 evidence 的结论一律不让出**
- 商家看不到 `[ev_xxx]` 哈希，但小二端 / 调试端能追溯

### 2.4 Review 兜底 —— 治高风险动作（不仅是幻觉，也是误操作）

- 调价 / 下架 SKU / 改活动配置 → 走 `agent/review.py` 状态机：候选 → 人工确认 → 执行
- 单纯展示型建议（换话术 / 改主图） → 自动出，无需人审
- 这是 §1 风险阈值的分流：**高风险动作不能完全交给模型**

## 3. 4 个节点的 Prompt 设计要点

四件套架起来后，每个节点还需要具体 prompt。按节点拆：

### 3.1 Checker · 体检节点

**任务**：异常识别

**幻觉风险**：低 —— 本质是数值比对

**Prompt 设计原则**：

- 异常检测**主体用规则引擎 + 阈值**，不用 LLM（数值比对让 LLM 做太贵也不准）
- 仅在需要"异常清单语义命名"时调 LLM（如把 `qianchuan_roi 0.8 vs 1.8` 命名为"千川投产比倒挂"）
- 调用前传入 Wiki 类目基线作为强约束，**不让 LLM 自己估基线**

### 3.2 Attributor · 归因节点

**任务**：MECE 拆维度 + 根因链 + 非数据信号检索

**幻觉风险最高的节点**：很容易瞎编"主播话术不行 / 主图不行"

**Prompt 设计 4 条硬约束**：

1. 每条根因必须挂 `evidence_refs`，没有就 `confidence < 0.5`，**Composer 过滤掉不给商家**
2. 候选根因必须是 Skill MD 列举的维度（流量 / 转化 / 商品 / 履约 / 用户 / 价格），**不能自创**
3. 跨指标耦合（如 CVR ↔ 主图 CTR）必须标 `cross_validation`，让 LLM 不能只看单个 metric 拍脑袋
4. 非数据信号（规则变动 / 频道升级）只从 Wiki `rule_changes.md` 检索，**不让 LLM 自己想"是不是有什么活动变化"**

### 3.3 Advisor · 行动建议节点

**任务**：从 Wiki 找资源挂钩 + 准入校验 + 性价比排序

**幻觉风险**：编造 `resource.url` / 准入门槛

**Prompt 设计 3 条硬约束**：

1. 每条 action 的 `resource.url` 必须**来自 Wiki `tool_db.md` / `activity_db.md` 的检索结果**，不能自造
2. 准入门槛字段（`eligibility.met`）由 Python 规则代码校验（不让 LLM 自评）—— 例如"体验分 ≥ 4.5"是 Python 比较 `profile.exp_score >= 4.5`，LLM 只生成解释文案
3. Wiki 没匹配资源 → 退化为"该方向待补"，**不强行编一个**

### 3.4 Composer · 报告组装节点

**任务**：把前 3 个节点的结构化输出渲染成 markdown 报告

**幻觉风险**：风格化润色时**改了事实**（把"跌 55.6%" 说成"跌 60%"以求口语化）

**Prompt 设计 4 条硬约束**：

1. **数字必须逐字保留**（不四舍五入、不改）
2. **仅做语言风格化**：工程话 → 老板话（参考 `00-项目说明书.md` §9.1 的 7 行对照表）
3. **不允许引入任何前序节点未提供的数字 / 资源 / 准入条件**
4. 走主窗口 `agent/nodes/diagnosis/_labels.py` 的 `metric_zh()` 字典 —— **确定性翻译**，不让 LLM 自由发挥

## 4. 真实迭代记录（不是理论推演）

主窗口 commit 节点：

- `bf1397e feat(diagnosis): wire real llm + enhance root cause / actions / opening`
- `24d6d58 feat(diagnosis): chinese labels + llm warning mute + summary polish`

副窗口 commit 节点：

- `f6a1ea8 refactor(demo): translate fields, strip ev tags, surface wiki refs`

迭代痕迹：

| 迭代                | 触发问题                                                      | 解决方案                                                                                                       | 来源                                   |
| ------------------- | ------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | -------------------------------------- |
| **v1**        | LLM 在根因句里直接输出英文 metric key（"qianchuan_roi 偏离"） | 加 `_labels.py` `metric_zh()` 中英对照字典；prompt 强约束"用中文 metric 名"；规则代码替换不让 LLM 自己翻译 | 主窗口 `24d6d58`                     |
| **v2**        | LLM 输出"可能 / 也许 / I think" 等保留词，给商家不靠谱感      | prompt 禁用保留词；置信度低就直接降 `confidence` 字段值，**不在文本里软化**                            | 主窗口 `24d6d58`                     |
| **v3**        | Composer 报告里残留 `[ev_xxxxxxxxxxxx]` 工程哈希标签        | demo 渲染层用 regex strip（侧道防御）；prompt 加"输出不允许出现 `[ev_` 开头标签"                             | 副窗口 `f6a1ea8`                     |
| **v4 计划中** | 根因句过短信息密度低（仅 10-15 字）                           | prompt 加"每条根因 30-80 字 + 必含数值证据 + 维度归属 + 影响范围"                                              | 待跑 08 验收用例 ground truth 对比后定 |
| **v5 计划中** | overlay 命中错位（货架店给自播建议）                          | Attributor 之前加 overlay 命中校验，错位时降级到通用 SOP                                                       | 待发现真实 bad case                    |

**从这些迭代里提炼的调 prompt 方法论**：

1. **先看 bad case，再改 prompt** —— 不要拍脑袋 "我觉得这样写更好"。08 验收用例就是给 prompt 迭代当 ground truth 用的
2. **改写 > 堆叠**：同一段约束反复加重不如改写一遍。LLM 对长 prompt 末尾的指令敏感度低
3. **能用 Python 校验的不让 LLM 校验**：`eligibility.met`、`metric_zh()` 翻译、`evidence_refs` 形状校验全用代码
4. **few-shot 用真实诊断结果**：不要造样例 —— 08 验收用例的 5 个 case 就是天然的 few-shot 池
5. **不要试图通过 prompt 教会模型领域知识**：把"商城精选准入条件"塞 prompt 是错的，要塞 Wiki，让 LLM 检索 —— 这跟 05 手册 Q5 教训完全一致（Diet 项目 VLM 幻觉饺子馅）

## 5. Bad Case 治理闭环

prompt 改了之后怎么知道有没有变好 / 变差？这套闭环：

```
商家点击建议 → 跳转资源
                ↓（资源 404 / 资源不匹配 / 跟商家实际不符）
              商家投诉 / 不采纳
                ↓
              Review 表标 "bad case"
                ↓
              小二抽检确认
                ↓
            写入 bad_cases.jsonl（按节点 + 类型分类）
                ↓
            每周回顾：是哪个节点出的？哪条 prompt 漏了？
                ↓
            分类施治：
              · 事实层错（基线 / 门槛 / URL）→ 改 Wiki MD
              · 策略层错（根因维度 / SOP）→ 改 Skill MD
              · 语言层错（话术 / 风格）→ 改 prompt
                ↓
            跑 08 验收用例回归 —— 5 个 case 是否还命中预期
                ↓
            合并到 main，更新 update.md
```

**为什么不直接 RL / 微调**：

| 选项                   | 这个项目不选的原因                                                                  |
| ---------------------- | ----------------------------------------------------------------------------------- |
| Reinforcement Learning | V1 阶段 bad case 量不够（5 case + 早期商家反馈 < 100 条），过拟合风险高             |
| Fine-tune              | 业务策略改太快（每周可能改 5-10 条 Wiki / Skill MD），fine-tune 一次 2-3 天，跟不上 |
| 大模型从头训           | 我们的核心知识是抖音电商规则，**这本来就在 Wiki 里能查到，没必要塞模型参数**  |

**选什么**：上面这套 "Wiki/Skill MD/Prompt" 三层热改 + 验收用例回归 —— **比训模型快 100 倍，业务可审计**。

## 6. 面试官追问预案（5 个）

| 追问                                       | 30 秒回答                                                                                                                                                              |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **"你 prompt 怎么写的？给我看一段"** | 把 `jingying_zhenduan.md` 拉给他看 —— 不是"我设计了一段 prompt"，是"这是 Skill MD 配置文件，业务侧能改"。**这本身就是产品观**：prompt 不是黑盒，是可审计配置 |
| **"怎么知道 LLM 有没有瞎编基线"**    | evidence_refs 必须有 `layer=wiki` 且 `sourceId` 指向真实 Wiki MD；没有 → confidence 降低 → Composer 过滤掉                                                       |
| **"如果 Wiki 没覆盖的指标怎么办"**   | 退化为 "暂无类目基线，仅同比" ——**绝不让 LLM 自由发挥**                                                                                                        |
| **"你们怎么评估 prompt 改完没变差"** | 跑 08 验收用例 5 个 case，对照"期望诊断 / 期望根因 / 期望建议"打分（V1 验收线：5 个至少过 4 个）                                                                       |

## 8. 一句话结论

> 抖店诊断 Agent 的防幻觉**不是 prompt 的事**，是 Skill MD + 双层 Wiki + evidence_refs + Review 四件套的事。
>
> Prompt 只是把这套架构结构化传达给 LLM 的最后一公里 —— 4 个节点 + 4 条调优原则 + 5 个真实迭代痕迹 + 1 套 bad case 治理闭环，组合起来才是这个项目能在 S4/S5 商家面前抬得起头的根本原因。
>
> 不靠模型自觉，靠产品架构 —— **这跟字节自家发品 Review 治理层（07 文档）是同一个产品理念**。
