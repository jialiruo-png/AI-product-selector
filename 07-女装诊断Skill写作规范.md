# 07 · 女装诊断 Skill 写作规范

> 给 6 个女装 overlay + 3 个通用 Skill MD 的统一写作约定。
> 任何 overlay 写偏了就会触发"同质化"短板回潮——这份规范是 V1 诊断质量的下限保障。

---

## 0. 一句话约束

**Skill MD 不是 prompt 文档，是"业务策略 + LLM 行为"的双重契约**——业务侧的人能读懂、改得动；LLM 侧能稳定按 schema 输出。

---

## 1. 文件层级与加载逻辑

```
agent/skills/
├── jingying_zhenduan.md        # 通用诊断入口
├── guiyin.md                    # 通用归因框架
├── xingdong_jianyi.md           # 通用行动建议
└── category_overlays/
    ├── nvzhuang_zibo.md         # 女装·自播店
    ├── nvzhuang_dabo.md         # 女装·达播店
    ├── nvzhuang_huojia.md       # 女装·货架店
    ├── nvzhuang_xindian.md      # 女装·新店冷启动期
    ├── nvzhuang_chengzhang.md   # 女装·成长期
    └── nvzhuang_jijie.md        # 女装·季节切换期
```

### 1.1 加载顺序（Loader 执行）

```
1. 加载 通用 Skill MD（jingying_zhenduan / guiyin / xingdong_jianyi）
2. 根据商家自动归属 → 加载对应 overlay（可多个）
3. 合并：overlay 覆盖通用的同名章节，未覆盖的章节继承通用
4. 输出最终 system prompt + SOP + 工具列表
```

### 1.2 一家店可能命中多个 overlay

例：一家月销 80w、入驻 60 天的女装自播店，会同时命中：
- `nvzhuang_zibo`（模式归属）
- `nvzhuang_xindian`（阶段归属，<90 天）

合并时按 **模式 overlay > 阶段 overlay > 季节 overlay** 的优先级。

## 2. Skill MD 标准结构（每个文件必须有的 9 个章节）

```markdown

## 0. 元信息
- skill_id: <唯一 ID>
- type: <main | overlay>
- applies_to: <商家归属条件，overlay 才有>
- version: <语义化版本号>
- last_updated: <YYYY-MM-DD>

## 1. 角色定义
<一句话定位这个专家是谁>

## 2. 核心目标
<这个专家要为商家解决什么具体问题>

## 3. 工作流程（SOP）
<分步骤的标准作业流程，每步必须可被 LLM 执行>

## 4. 工具依赖
<列出可调用的工具名，必须存在于 agent/tools/ 注册表>

## 5. 输入 Schema
<期望的输入字段，JSON 结构>

## 6. 输出 Schema
<期望的输出字段，JSON 结构，包含证据引用>

## 7. 类目/场景特化（仅 overlay）
<本 overlay 相比通用 SOP 的差异化要点>

## 8. 示例（输入-输出 1-2 组）
<让 LLM Few-Shot 学习的标准案例>

## 9. 反例与红线
<不该这么做的反例 + 必须规避的红线>
```

## 3. 字段写作强制约束

### 3.1 SOP 章节（§3）必须

- **每步动作必须带"决策点"**：什么条件下做什么；不能写"分析数据"这种空话
- **每步动作必须可工具化**：要么调具体工具、要么调 LLM 子任务，不能是"由专家判断"
- **每步必须有 fallback**：缺数据/工具失败时怎么走

✅ 正确写法：
```markdown
1. 调用 fetch_shop_metrics 获取近 7 天 GMV、UV、CVR、退款率
2. 若 CVR < 类目基线 × 0.8 → 进入"转化率归因"分支
   - 调用 fetch_category_baseline(类目=女装, 模式=自播)
   - 调用 guiyin 专家，输入维度=[流量精准度, 货品力, 价格, 素材]
3. 若 CVR 正常但 GMV 跌 → 进入"流量归因"分支
4. 数据缺失（任意指标返回 null）→ fallback 走"通用体检模板"
```

❌ 错误写法：
```markdown
1. 分析店铺数据
2. 找出问题
3. 给建议
```

### 3.2 输出 Schema（§6）必须

每条结论字段都要带：
- `claim`：结论文本
- `evidence_refs`：至少 1 个 `KnowledgeEvidenceRef`
- `confidence`：0-1 的置信度
- `data_sources`：数据来源（raw / wiki / llm_inference）

示例：
```json
{
  "diagnosis": [
    {
      "claim": "你的连衣裙 SKU 转化率 1.3% 显著低于女装自播类目均值 2.4%",
      "evidence_refs": ["#shop_metrics_001", "#wiki_category_baseline_dress"],
      "confidence": 0.92,
      "data_sources": ["raw", "wiki"]
    }
  ]
}
```

### 3.3 行动建议（仅 xingdong_jianyi.md 和相关 overlay）必须

每条建议必须挂**至少一个**可执行资源：
- `activity_id`：可报名的具体活动 ID
- `tool_url`：可一键跳转的官方工具
- `template_id`：可复用的模板（千川计划 / 主图模板 / 话术模板）

❌ 空建议禁出：
- "建议优化主图"
- "建议报名活动"
- "建议加强客服"

✅ 必须这样写：
- "建议用智能图片工具重做主图（参考模板 #M0247），一键跳转：tool://aigc_image?template=M0247"
- "建议报名「618 女装预热活动」（活动 ID #618_nvzhuang_yure），准入门槛=体验分 4.5+，你当前 4.3，差 0.2"

### 3.4 反例与红线（§9）必须

每个 Skill MD 至少写 3 条红线。常见红线：
- 不许给空建议
- 不许虚构活动 ID / 工具入口
- 不许在数据不足时下确定性结论（必须降级为"猜测"）
- 不许涉及超出商家权限的建议（如直接改价、直接联系小二）

## 4. 三个通用 Skill MD 的差异定位

| Skill | 角色 | 核心 SOP 节点 | 输出物 |
|---|---|---|---|
| **jingying_zhenduan.md**（经营诊断专家） | **入口调度** | 体检全维度 → 识别异常 → 决定是否进入归因 | 异常清单 + 严重度 |
| **guiyin.md**（归因专家） | **根因推导** | MECE 拆维度树 → 跨场景数据交叉 → 定位根因 | 根因链路 + 置信度 |
| **xingdong_jianyi.md**（行动建议专家） | **可执行翻译** | 根因 → 可执行动作 → 挂资源 → 排优先级 | 建议清单（含资源挂钩） |

**注意**：三者**不能职责越界**。诊断专家不许给建议，归因专家不许出动作，建议专家不许跑数据归因——边界清晰才能让 LLM 稳定。

## 5. 六个女装 overlay 的差异化要点

每个 overlay 的 §7 必须明确写以下三点：

1. **本场景商家典型痛点**（与通用诊断的差异）
2. **本场景该重点看的 3-5 个指标**
3. **本场景常用的 5-10 个建议模板**（挂资源后可直接产出）

### 5.1 nvzhuang_zibo.md（自播店）

- **典型商家**：白牌 / 工厂店 / 自播为主，直播间贡献 60%+ GMV
- **重点指标**：直播间人均停留时长、转粉率、UV 价值、主播话术转化、上链接 CTR
- **建议模板**：直播间话术优化 / 主图重做 / 福袋节奏 / 千川直播投流

### 5.2 nvzhuang_dabo.md（达播店）

- **典型商家**：品牌 / 精品 / 靠达人带货，达人贡献 50%+ GMV
- **重点指标**：达人坑位 ROI、达人佣金率、达人选品命中率、合作 CR
- **建议模板**：达人筛选 / 坑位提报 / 品牌专场 / 精选联盟优化

### 5.3 nvzhuang_huojia.md（货架店）

- **典型商家**：搜索 + 商城为主，老款长尾，无直播
- **重点指标**：搜索曝光 / 点击率、商品标题命中关键词、评价分、商城频道占比
- **建议模板**：标题优化 / 评价管理 / 商城精选报名 / SEO 关键词

### 5.4 nvzhuang_xindian.md（新店冷启动期，≤90 天）

- **典型商家**：体验分 0 或低、流量靠平台扶持、无历史数据基线
- **重点指标**：突破 0 分进度、新手任务完成率、首单转化、店铺装修完整度
- **建议模板**：突破 0 分任务清单 / 首单破冰活动 / 新手扶持流量入口 / 必报活动

### 5.5 nvzhuang_chengzhang.md（成长期，月销 30w-300w）

- **典型商家**：有数据但增长卡壳、千川/商城多渠道、需要找新增量
- **重点指标**：流量来源多样性、千川 ROI、新品动销率、复购率、UV 价值
- **建议模板**：千川计划优化 / 商城精选 / 新品提报 / 老客复购召回

### 5.6 nvzhuang_jijie.md（季节切换期）

- **典型商家**：夏装/秋装切换、应季款风险高、库存压力大
- **重点指标**：滞销库存占比、应季款备货充足度、应季关键词搜索趋势
- **建议模板**：滞销款清仓 / 应季款加投 / 应季活动报名 / 库存预警

## 6. Skill MD 示例片段（节选自 `nvzhuang_zibo.md`）

```markdown
# 女装·自播店 · nvzhuang_zibo

## 0. 元信息
- skill_id: nvzhuang_zibo
- type: overlay
- applies_to:
    category: 女装
    mode: 自播
    condition: 直播 GMV 占比 ≥ 60%
- version: 1.0.0
- last_updated: 2026-06-04

## 1. 角色定义
你是抖音电商女装自播店的资深运营顾问。你专门服务直播间是主流量、价格敏感、白牌/工厂店为主的商家。

## 2. 核心目标
帮商家定位"为什么直播间不出单"的根因，给出可执行的直播间话术、主图、福袋节奏优化建议，并挂具体资源。

## 7. 类目/场景特化

### 典型痛点
- 直播间人均停留 <30 秒就跳出
- 转粉率低（<2%），下播即流失
- 主图与直播间画面割裂，进直播间不转化
- 千川直播投流 ROI 倒挂

### 重点指标（覆盖通用诊断后必查）
- 直播间人均停留时长（健康基线：>60 秒）
- 转粉率（健康基线：>3%）
- UV 价值（直播间 GMV / 进场 UV）
- 上链接 CTR（健康基线：>15%）
- 主播话术转化关键节点

### 建议模板（每条都挂资源）
- "建议优化直播间欢迎话术"（template_id: live_welcome_v2）
- "建议调整福袋节奏到开播 5 分钟"（template_id: fudai_rhythm_001）
- "建议报名「自播扶持流量包」"（activity_id: zibo_liuliang_2026）
- "建议重做主图突出价格锚定"（tool_url: aigc_image?template=zibo_price_anchor）

## 9. 反例与红线
- ❌ 不许建议商家"换主播"——超出运营建议范围
- ❌ 不许在直播 GMV < 20% 的店上跑本 overlay（应走货架/达播 overlay）
- ❌ 不许在通用诊断没跑完前给本场景特化建议
```

## 7. 写作时的协作流程

1. **先写通用 Skill MD（3 份）** → 跑 mock 验证可加载、可触发
2. **再写女装 overlay（6 份）** → 一个一个写，每写完一个跑 mock 看合并效果
3. **每个 overlay 必须配 1 组验收用例**（写在 `08-验收用例-女装真实场景.md`）
4. **修改 overlay 前先看是否影响其他 overlay 的合并结果**

---

## 8. 验收口径

每个 Skill MD 完成的标准：
1. 9 个标准章节齐全
2. SOP 每一步都有决策点 + 工具 + fallback
3. 输出 schema 有 evidence_refs 字段
4. 建议（如有）都挂了资源
5. 至少 3 条红线
6. 至少 1 组完整示例（input → output）
7. Loader 能成功加载并 print system prompt 验证

---

## 9. 未来扩展指引

V1 验证有效后扩展到其他类目时：
- 复用本规范的 9 章节结构
- 直接 fork `category_overlays/nvzhuang_*` 改类目特化部分
- 通用 Skill MD（jingying_zhenduan / guiyin / xingdong_jianyi）原则上不动
- 新增类目（如食品）走类似的"模式 × 阶段 × 季节"三维切分

---

**最后**：这份规范本身也是 v1.0，写到第 3-4 个 overlay 时如果发现哪条约束不合理，**先在这里改**，再批量回写已写的 overlay。规范优先于实现。
