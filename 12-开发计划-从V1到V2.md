## 0. 一句话路线

> **M0 商家访谈（立即并行）→ M1 质量与治理底座（评测 / 可观测 / Review 接线）→ M2 产品价值升级（三视图话术 / 跨子图协同 / 成本三杠杆）→ M3 按 ROI 探索（直播闭环 / 双视角选品 / MCP 化）。**
>
> M1 是其余一切的地基：没有自动化评测，M2 的每次 prompt / Skill MD 改动都无法回归；没有可观测，成本三杠杆无从度量。

## 3. M0 — 商家访谈（00 §9.4，优先级最高的"非代码任务"）

**为什么排第一**：9.1 指标白名单不能自己拍脑袋；9.2"消费者直觉"假设、9.5"场次复盘"需求强度，都要访谈验证。越早被颠覆代价越低。

| 项 | 内容 |
|---|---|
| 样本 | 5-10 个抖店女装 S4/S5 店主（校友 / 朋友圈 / 小红书 / 商家社群） |
| 方法 | 给看 Streamlit demo（case_1 咖喱生活），问三个核心问题（最在意哪三个数字 / 哪条建议会真做 / 多久推一次） |
| 产出 | ① 指标白名单 ② 指标黑名单 ③ 推送频次建议 → 直接喂给 M2-1 |
| 负责人 | 贾丽婼主导（诊断子图），甘华梁可顺带验证 9.2 选品偏好假设 |
| 风险 | access 难、样本偏差 → 哪怕只有 3 个有效访谈，也比 0 个强；结论标注样本量 |

## 4. M1 — 质量与治理底座（核心里程碑，先做）

### M1-1 评测 harness（G1，最高优先级）

- **文件**：新增 `tests/eval_diagnosis.py`、`tests/eval_insight.py`、`tests/golden/`（期望输出快照）
- **做什么**：
  1. **断言层（不调 LLM，秒级）**：5 个诊断 case 在 mock 模式下逐一跑 `run_diagnosis`，断言——命中 overlay 正确、核心问题 metric 正确、anomalies 数量符合区间、每条 action 有 `resource.url`（红线）、每条结论挂 evidence_refs（红线）、报告非空。选品子图同理断言 9 节点全到齐 + 证据数 ≥ 基线。
  2. **LLM-as-judge 层（带 key 才跑，可选）**：把真 LLM 版报告 + `08-验收用例` 的期望四段对照喂给裁判模型打 1-5 分，输出对比表——把批次 10 的"主观打分"自动化。
  3. 统一入口：`python3 -m tests.eval`（或 Makefile），把 CLAUDE.md §9 的 V1 验收五条全部脚本化。
- **验收**：故意改坏一个 overlay 阈值 → eval 立刻红；恢复 → 绿。全量断言层 < 30 秒。
- **约束**：纯新增文件，不动任何禁区；mock 模式零 key 可跑（CI 友好）。
- **负责人**：诊断 case 贾丽婼 / 选品 case 甘华梁；harness 骨架谁先动手谁写，另一方 review。

### M1-2 Review 状态机接线（G2）

- **文件**：`agent/nodes/diagnosis/advisor.py`（贾丽婼独立资产）；`agent/state.py` 加 `review_items` 字段（共用资产，**需双方确认**）
- **做什么**：
  1. 给 action 定义风险分级：**高风险**（停投千川 / 下架 SKU / 改价——做错有真金白银损失）vs 低风险（换话术模板 / 报名活动）。
  2. 高风险 action 调 `ReviewQueue.submit(payload)` 进候选，报告中标注「⏳ 待确认」；低风险直接输出。
  3. CLI 加 `--approve <review_id>` 演示确认流（Streamlit demo 加确认按钮，副窗口任务）。
  4. （可选，LangGraph 后端）用 checkpointer + `interrupt` 实现真正的"暂停等人审、确认后续跑"——这是 LangGraph 原生 human-in-the-loop 范式，MiniGraph 兜底路径保持现状（队列模式）即可。
- **验收**：case_1 的"今晚停投千川"类建议进 Review 队列而非直接输出；approve 后状态机日志可查。
- **价值**：03 PRD §1.4 的"人审边界"从口号变成可演示的产品行为——这也是面试 / 评审最常被追问的治理问题。

### M1-3 运行可观测落库（G3）

- **文件**：`agent/llm.py`（共用资产，**需双方确认**）、新增 `agent/observability.py`、复用 `db.py` 模式新增 `agent_runs` 表
- **做什么**：
  1. `chat_json / chat_text` 回传并累计 `usage`（prompt / completion tokens），挂到 state 的 `_llm_usage`。
  2. 每次 run 结束落一行：`run_id / subgraph / backend / 节点耗时明细 / LLM 调用次数 / tokens / 估算成本（元）/ evidence 计数 / 是否 mock`。
  3. CLI 报告尾部打印一行成本摘要：`[cost] LLM 12 次 · 14.3K tokens · ≈0.04 元`。
- **验收**：跑 5 个 case 后能 SQL 查出每 case 成本对比；00 §10.1 的"单次诊断 10-15K token"从推演值变成实测值。
- **价值**：M2-3 成本杠杆的所有"降了多少"都要靠这张表说话；也直接回答字节面试"token 成本算过吗"。

### M1-4 提交流程固化（半天）

- 把 CLAUDE.md §4.1 的两条命令 + M1-1 eval 包成 `make check`（或 `scripts/check.sh`）；有 GitHub 仓库则加 Actions 跑 mock 模式断言层。
- **验收**：`make check` 一条命令 = compileall + 双子图 mock + eval 断言层。

## 5. M2 — 产品价值升级

### M2-1 商家话术三视图（G5，对应 00 §9.1）

**前置**：M0 访谈白名单到位；M1-1 eval 护航（话术改动跑回归）。

- **文件**：新增 `agent/wiki/industry/talk_mapping.md`（话术映射表，**配置即知识**——不写死在代码里）；`agent/nodes/diagnosis/composer.py` 改造（贾丽婼独立资产）；`agent/nodes/diagnosis/_labels.py` 扩展
- **做什么**：
  1. 按访谈白名单写 10-15 条「工程话 → 老板话」映射 + 6 条话术原则（00 §9.1 表格已起草 7 条），落到 `talk_mapping.md`，loader 零改动可读。
  2. Composer 增加渲染参数 `view ∈ {owner, ops, team}`：
     - **owner**（默认）：1 句问题 + 3 条带「花不花钱 / 几天见效」的动作 + 1 条提醒；**不出现** metric 名 / 置信度 / 偏离百分比；比例一律换算成钱（"少卖了 16 万"）。
     - **ops**：现 Streamlit 形态（漏斗 + 根因链 + 建议表）。
     - **team**：完整指标 + SKU 维度。
  3. LLM 只做风格化重写，数字逐字保留（沿用 10-幻觉治理 §3 Composer 四条约束）；估算类数字强制带"估算/预估"前缀。
  4. CLI 加 `--view`，Streamlit 加视图切换（副窗口）。
- **验收**：case_1 老板版报告通过"零 metric 名"扫描（正则断言进 eval）；ops / team 视图与现版一致无回归；访谈对象回看老板版，至少 3/5 表示"能看懂、知道要做什么"。
- **关键约束**：input schema 不动——Checker 仍吃全量罗盘字段，只动输出层。

### M2-2 跨子图协同：诊断触发选品（G6，对应 00 §9.3）

- **文件**：`agent/state.py` 加 `child_subgraph_results`（共用资产，**需双方确认**）；`agent/nodes/diagnosis/advisor.py`；`agent/hub.py` 加跨子图中转（共用资产，**需双方确认**）
- **做什么**：
  1. Advisor 识别根因 dimension ∈ {库存, 选品, 应季} 时，经 Hub 中转调用 `run_insight`（不直接 import 对方子图——保持"图之间不直接通信"）。
  2. 选品结果裁剪为 Top-3 SKU 建议挂进 `actions`，证据引用标注来源子图。
  3. **错误隔离红线**：选品子图超时 / 失败时，诊断报告照常产出（advisor 已有 fallback 范式，沿用）；mock 模式下选品子图也走 mock。
- **验收**：case_5（夏树·季节切换）的"备秋装"建议从方向性话术变成"建议上这 3 款秋装（含价格带 / 竞品痛点规避）"；选品子图被 kill 时诊断报告无损输出。
- **价值**：这是"共底座、独立编排、Hub 路由"叙事的**真正落地**——目前两个子图只是平行共存，做完这条框架整合价值才被验证。

### M2-3 成本三杠杆落地（G4，对应 00 §10.3）

**前置**：M1-3 可观测——没有实测基线，"降了多少"无从谈起。

- **杠杆 A · 分层触发漏斗**：新增 `agent/nodes/diagnosis/prescreen.py`（纯规则、零 LLM）：指标全绿的商家直接出"本周健康，无需诊断"短报告，不进 4 节点链。新增 1 个"健康商家" mock case 验证。→ 对应全量场景砍 95% LLM 调用。
- **杠杆 B · 按节点模型路由**：`llm.py` 的 `chat_json/chat_text` 加 `tier` 参数（`cheap / standard / best`），`.env` 配三档模型名；attributor 主因解释走 standard、advisor "为什么做"走 cheap、composer 开篇走 best。**共用资产改动，需双方确认。**
- **杠杆 C · prompt 缓存友好重排**：所有 LLM 调用把**静态前缀**（Skill MD SOP + overlay + Wiki 条目）放 system / 消息最前，动态商家数据放最后——provider 侧前缀缓存即可生效；DeepSeek 已默认开 context caching，零代码成本先吃到。
- **验收**：M1-3 的 `agent_runs` 表对比改造前后：单次诊断 token / 成本下降可量化（目标：成本 -40% 以上且 eval 全绿）。
