## 2. 需求背景

### 2.1 平台事实：90% 中小商家没有人工小二服务

依据《小二职责与痛点》文档梳理：

- **人力配比硬约束**：1 个类目运营小二对接 300-800 家中小商家 + 几十家头部 KA。60% 工时被头部 KA 占用，其余时间处理风控事故。
- **90% 平稳经营、数据偏弱的中小商户全年不会收到 1 次小二主动私聊。**
- 这不是小二渎职，是平台制度性结果——人力精力锚定 GMV 贡献，中小单店对大盘影响低，**不可能配人工一对一帮扶**。

### 2.2 AI 接管中小商家了，但接管质量很烂

当前抖店 AI 助手已经覆盖了 8 个数据分析工具（成交/退款/体验分/同行/异动/竞品/流量/经营总结），但服务中小商家时存在 4 个致命短板：

| 短板 | 表现 | 商家感受 |
|---|---|---|
| **① 报问题不找根因** | "转化率下跌，建议优化详情" | 我知道跌了，我想知道为什么跌 |
| **② 同质化套话** | 服饰/食品/家居套同一套话术；自播店和货架店给一样的建议 | AI 根本不懂我做的是什么 |
| **③ 缺资源配套** | "去报活动拉流量" | 报哪个？我够格吗？怎么才够格？ |
| **④ 不处理非数据信息** | 规则变动、类目玩法调整、活动门槛变化 → AI 完全不知道 | 我数据没问题但流量莫名其妙跌了 |

### 2.3 后果：恶性循环

```
数据下滑 → AI 空话式指导 → 落地无效 → 营收继续低 →
商家失去运营信心 → 放弃店铺 → 平台中小活跃度下滑
```

### 2.4 战略价值

平台明知道这是问题，但短期不可能扩招小二。后续路径是**精细化 AI 迭代 + 分层社群运营**——本 PRD 正是这条路径的产品落地。

**做好这件事的两层价值**：
1. **直接价值**：让 S4/S5 商家真正能从 AI 诊断里获得有效经营优化，提升中小盘活跃度
2. **间接价值**：自动把"AI 已诊断 + 商家已采纳 + 指标已回升"的店打标推到小二 S3 潜力池——给小二的潜力商家挖掘提供高质量货源

## 3. 预期收益

### 3.1 直接业务收益（推演口径）

| 指标 | 推演值 | 推演依据 |
|---|---|---|
| **诊断建议采纳率** | ≥ 35% | 对标抖店 AI 助手"立即合作达人/去铺货"类按钮的当前点击率基线（业内 15-20%），叠加根因型建议的相关性提升 |
| **建议执行率**（采纳后真去做） | ≥ 60% | 因为每条建议挂具体可执行入口（活动 ID / 主图工具 / 千川计划），降低执行摩擦 |
| **指标回升率**（执行后 14 天对应指标改善） | ≥ 25% | 类目+模式定制 SOP 比通用建议命中率显著高 |
| **覆盖商家数** | 300w+ | S4/S5 商家估算盘子 |
| **MAU 渗透** | ≥ 8%（首月） | 主动推送+对话双入口拉动渗透 |

### 3.2 间接战略收益

- **小二端潜力池货源**：每月由本 Agent 培育出 X 万家"已采纳建议+指标改善"的店推送给小二，提升小二 S3 池命中率
- **平台规则透传效率**：行业 Wiki 沉淀规则变动后，中小商家平均规则触达延迟从"2-4 周"降至"24 小时内"
- **小二经验沉淀**：老小二沟通话术 / 类目玩法以 Skill MD 形式落地，新小二培养周期缩短

## 4. 用户与场景

### 4.1 主用户：S4/S5 在运营中小商家

| 用户特征 | 商家诉求 |
|---|---|
| 已开店 ≥ 30 天，有基础经营数据 | 看到罗盘指标动了，不知道为什么 |
| 体验分 < 4.7 或近期波动大 | 想优化但不知道改哪里 |
| 月销 < 50w，不在 KA 名单 | 拿不到人工小二，又看不懂罗盘 |
| 无专职运营，老板/夫妻档亲自做 | 时间稀缺，要"5 分钟看完+1 键执行" |

### 4.2 次用户：类目运营小二（间接）

| 小二诉求 | Agent 给到 |
|---|---|
| S3 潜力池缺货源 | 推送"已采纳建议+指标改善"的中小商家 |
| 不知道哪些 S5 商家在做积极改善 | 每周给小二一份"潜力升级候选名单" |

### 4.3 三个核心场景

**场景 A · 主动诊断（推送式）**
- 触发：商家关键指标偏离基线（GMV/UV/CVR/体验分异常）
- 入口：抖店首页/AI 助手对话首屏卡片
- 输出：3 分钟可读的体检报告（1 个核心问题 + 根因链 + 3 个可执行动作）

**场景 B · 主动询问（对话式）**
- 触发：商家问"为什么我这周销量跌了"
- 入口：AI 助手对话框
- 输出：同 A，但围绕用户具体问题展开

**场景 C · 定期复盘（周报）**
- 触发：每周一自动推送
- 输出：上周经营摘要 + 趋势归因 + 本周建议优先级 + 已采纳建议的效果跟踪

## 5. 产品方案

### 5.1 核心能力栈（与 V2 PRD 四层架构对齐）

```
┌────────────────────────────────────────────────┐
│ L4  触达层    主动推送 + 对话 + 周报 + 小二端候选名单 │
├────────────────────────────────────────────────┤
│ L3  建议层    可执行动作生成（挂具体活动 ID/工具入口） │
├────────────────────────────────────────────────┤
│ L2  归因层    根因链路推导（多维 MECE 拆解 + 跨场景）   │
├────────────────────────────────────────────────┤
│ L1  数据层    商家罗盘 + 行业 Wiki + 商家 Wiki + 工具调用 │
└────────────────────────────────────────────────┘
```

### 5.2 五步诊断闭环（核心工作流）

```
①  体检    自动跑全维度健康度扫描，识别异常指标 + 严重程度
    ↓
②  归因    针对每个异常拆 MECE 维度树，定位根因（货品/流量/转化/履约）
    ↓
③  建议    把根因翻译成可执行动作，每条动作挂具体入口（活动 ID / 工具 / 计划）
    ↓
④  执行    商家一键跳转去做 / 拒绝建议
    ↓
⑤  复盘    7-14 天后回看指标变化，AI 自评建议有效性 → 反哺 Skill MD
```

### 5.3 多专家 Agent 矩阵（V1 只上 3 个，避免过度设计）

| 专家 | Skill MD 文件 | 核心职责 |
|---|---|---|
| **经营诊断专家** | `agent/skills/jingying_zhenduan.md` | 入口节点：识别异常 + 决定要不要进入归因链路 |
| **归因专家** | `agent/skills/guiyin.md` | MECE 拆维度树 + 根因定位 + 跨场景数据交叉 |
| **行动建议专家** | `agent/skills/xingdong_jianyi.md` | 把根因翻译成可执行动作 + 挂资源 + 排优先级 |

**V2 可扩**（不在 V1 范围）：客服话术专家、千川投流专家、库存补货专家、合规整改专家。

### 5.4 Skill MD 设计原则（关键差异化）

#### 5.4.1 类目 × 模式矩阵化（解决"同质化"短板）

每个专家 Skill MD 不是一个 prompt，而是**类目 × 店铺模式** 的二维矩阵：

```
agent/skills/jingying_zhenduan.md             # 通用入口 SOP
agent/skills/guiyin.md                         # 通用归因框架
agent/skills/xingdong_jianyi.md                # 通用建议生成 SOP

agent/skills/category_overlays/
  ├── nvzhuang_zibo.md        # 女装·自播店 SOP（强短视频+主图）
  ├── nvzhuang_huojia.md      # 女装·货架店 SOP（强搜索+评价）
  ├── shipin_zibo.md          # 食品·自播店 SOP（强信任+复购）
  ├── jiaju_dabo.md           # 家居·达播店 SOP（强达人+客单价）
  └── ...                     # V1 先做 6 个高频组合，后续扩
```

Loader 加载时**通用 SOP + 类目 overlay 合并**，前者定框架后者改细节。

#### 5.4.2 资源配套数据库（解决"缺资源"短板）

在 `agent/wiki/industry/` 下新增：

- `activity_db.md`：当前可报活动清单（活动 ID / 准入门槛 / 报名截止 / 历史命中率）
- `tool_db.md`：可一键跳转的官方工具列表（智能发品 / 短视频生成 / 千川计划模板等）
- `rule_changes.md`：近 30 天平台规则/玩法变动（解决"非数据信息"短板）

**关键约束**：行动建议专家产出的每条建议**必须**挂至少一个具体资源（活动 ID 或工具入口），不允许空建议。

### 5.5 双层 Wiki 内容增量（在甘华梁现有基础上扩）

| Wiki 层 | 甘华梁已有 | 本 PRD 新增 |
|---|---|---|
| **行业 Wiki**（只读种子） | 选品打分口径、价格带 SOP | 类目健康度基线、活动数据库、工具数据库、规则变动日志 |
| **商家 Wiki**（私有 overlay） | 商家选品偏好 | 商家历史诊断记录、已采纳/拒绝的建议、动作执行结果反馈 |

**冲突治理**（V2 PRD 6 种语义关系直接复用）：
- 行业基线说"CVR < 2% 需预警" vs 商家自述"我这类目本来就 1%" → 走 `scope_differs`（适用范围不同）逻辑：诊断时给出"按类目基线偏低，按你的店历史正常"的双口径表达。

### 5.6 Review 治理边界（什么需要人审）

| 建议类型 | 是否需要 Review | 说明 |
|---|---|---|
| 数据解读类（"你的 CVR 低于类目均值"） | ❌ 直出 | 事实陈述无风险 |
| 优化建议类（"优化主图、补 SKU"） | ❌ 直出 | 低风险动作 |
| 资源申请类（"建议报 XX 活动"） | ❌ 直出但挂"是否报名"商家自主决策 | 商家点了才生效 |
| 高敏感动作（"建议改价至 X 元"） | ✅ 商家二次确认 | 价格调整影响大 |
| 平台资源争取（"建议申请流量绿通"） | ✅ 小二审核 | 涉及平台资源分配 |

## 8. 评估指标（北极星 + 过程）

### 8.1 北极星（产品级）

**采纳→执行→指标改善的全链路转化率** = (建议被点击 × 执行 × 指标真改善) / 总建议数

### 8.2 过程指标

| 阶段 | 指标 | V1 目标 |
|---|---|---|
| 诊断质量 | 异常识别准确率（人工抽检） | ≥ 80% |
| 归因质量 | 根因被商家/小二认可率 | ≥ 70% |
| 建议质量 | 建议被点击率（采纳率） | ≥ 35% |
| 建议落地 | 点击后 7 天内执行率 | ≥ 60% |
| 业务回报 | 执行后 14 天对应指标回升率 | ≥ 25% |
| 用户感受 | 诊断报告满意度（👍/👎） | ≥ 75% 满意 |

### 8.3 反向监控

- 错误建议占比（事后人工/数据校验） ≤ 5%
- 商家投诉率（"AI 误导了我"） ≤ 1%
- Token 成本/商家/月（成本预算守门）

## 10. 风险与边界

| 风险 | 缓解 |
|---|---|
| 类目 overlay 颗粒度不够 → 诊断仍同质化 | V1 先做 6 个高频组合验证有效性，再批量扩 |
| 活动数据库需要持续维护 | 行业 Wiki 接管，运营同学按周更 MD；版本化追踪 |
| 错误建议导致商家亏损 | 高敏感动作（改价）走二次确认；建议附"参考依据"+"风险提示"双段 |
| Token 成本失控 | 节点级预算守门（沿用甘华梁 `max_cost_calls` 模式）；分档调用（参考 Soul 缘分档案分档 LLM 经验） |
| 与商家自身判断冲突 | 用 V2 PRD 的 `scope_differs / contradicts` 治理矩阵处理 |

## 14. 一句话总结（面试 / 评审用）

> 90% 的中小商家拿不到人工小二，被 AI 接管了但接管质量很烂。本 PRD 不是再做一个"小二副驾"——小二的 AI 排序已经够好——而是**直接补齐 AI 接管中小商家时的 4 个具体短板**：报问题不找根因、不分类目套话术、不挂可执行资源、不处理规则变动。技术上沿用多专家 Agent + Skill MD + 双层 Wiki 三件套，与同事冷启动选品子图共底座、独立子图，通过 Hub 路由分发不同意图。
