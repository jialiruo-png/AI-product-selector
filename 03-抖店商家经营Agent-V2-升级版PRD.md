# 抖店「商家经营 Agent」V2 升级版 PRD

> 候选人：贾丽婼 · 面试岗位：字节跳动-抖音电商-商家产品实习生
> 本版相对 V1 的升级：从单点"AI 客服 Agent"扩展为**平台级商家经营 Agent 操作系统**
> 设计参考：业界第三方电商 AI SaaS（多专家矩阵 + Skill MD + Wiki 双层治理 + Review 闭环）

---

## 0. 一句话产品定位

将抖店从"商家自服务后台"升级为"**商家经营操作系统**"——商家不再被动看罗盘、自己想策略，而是由**15 个 AI 专家 Agent + 抖音电商行业 Wiki + 商家私有知识库**共同驱动，覆盖"新品冷启动 → 存量经营 → 内容场 → 决策辅助"全链路，把"目标→诊断→规划→执行→复盘"的全经营闭环自动化。

## 1. 设计哲学（4 个核心理念）

### 1.1 多专家 Agent 矩阵（而不是单一大 Agent）

业界共识：电商运营是高度专业化的领域，**单个大 Agent 工具数量 200+ 必然选错工具**。正确做法是按运营专业领域拆分专家，每个专家有自己的：
- system prompt（专业知识）
- 工具集（限定调用范围）
- SOP 标准作业流程

**借鉴依据**：我在 Relty Wardrobe 项目实测 Hub Agent → Fashion Agent 的 Handoff 模式，意图识别准确率 95%+，验证了"路由 Agent + 专家 Agent"双层架构的稳定性。

### 1.2 Skill MD 范式（配置即技能）

每个专家 Agent 不是硬编码的代码模块，而是一份**人可读、业务可改的 Markdown 文件**：
- 角色定义
- 核心目标
- SOP 标准作业流程
- 工具依赖
- 输入输出 schema

**好处**：业务运营专家可以直接改 MD 优化策略，不需要研发介入。这是 OpenClude / Anthropic Skills 思想的工程落地。

### 1.3 行业 Wiki + 商家 Wiki 双层架构

```
┌─────────────────────────────────────────┐
│ 抖音电商行业 Wiki（平台预编译，只读）         │
│ - 抖音电商指标定义（GPM/DSR/体验分等）      │
│ - 各类目运营 SOP / 爆款元素库             │
│ - 千川/百应/巨量学官方策略库              │
│ - 平台规则与违规案例                     │
└─────────────────────────────────────────┘
                  ↓ 继承 / 覆盖
┌─────────────────────────────────────────┐
│ 商家私有 Wiki Overlay（租户私有，可增长）    │
│ - 自有 SKU 详情/卖点/库存                │
│ - 历史经营数据/客服话术/退换货政策          │
│ - 商家自定义运营 SOP                    │
│ - AI 高质量回答的回流沉淀                 │
└─────────────────────────────────────────┘
```

**为什么这是关键**：普通 RAG 是"每次问题临时检索"，知识不沉淀；Wiki 双层结构能让"每次商家提问 + AI 回答 + 人工确认"持续编译进 Wiki，**知识复利**。

**冲突治理矩阵**：商家私有知识和行业 Wiki 不可避免会冲突（指标口径、素材规范、动作优先级、组织流程）。系统按 6 种语义关系自动归类，分别走不同处理路径：

| 语义关系 | 含义 | 处理策略 |
|---|---|---|
| supports / duplicates | 商家知识与行业知识互证 | 合并证据，置信度提升 |
| refines | 商家在行业规则上做细化 | 商家覆盖生效，保留行业来源指针 |
| scope_differs | 适用范围不同（如类目/价位段） | Agent 回答时明确标注适用范围，不判定冲突 |
| contradicts | 真冲突 | 进入人工 Review，未裁决前 Agent 默认走行业版本 |
| supersedes | 新文档主张替代旧规则 | 需附版本/时间/权威性证据，否则不允许覆盖 |

> **借鉴依据**：转化组方案的语义单元（SemanticUnit）治理思想直接落地。这避免了"商家说什么 AI 就改什么"的单边覆盖风险，也避免了"AI 死守行业默认知识"的僵化。

### 1.4 Review 治理（人审边界明确）

AI 永远不直接改写商家关键数据。任何高敏感动作（改价、退款、上下架）必须经过 Review 状态机：
- **Candidate**：AI 输出候选方案
- **Pending Review**：等待商家/小二审阅
- **Approved**：人工通过
- **Executed**：执行并记录审计日志
- **Rejected**：驳回 + 反馈给 AI 学习

**借鉴依据**：我在亚美特 OMS 项目实现了 9 状态 + 11 动作的 ApprovalLog 状态机，理解 B 端业务中"AI 建议必须有人审核"的合规边界。

## 2. 抖店官方版的差异化（vs 第三方 SaaS）

这是关键判断点：**为什么抖店做这个比第三方 SaaS 做有不可替代的优势？**

| 维度 | 第三方 SaaS（如小龙虾等） | 抖店官方内嵌（本 PRD） |
|---|---|---|
| **数据接入** | 通过 OpenAPI 抓取，延迟高、有限速 | 原生 API 直连，实时、无限制 |
| **内容场协同** | 做不到（抖音内容数据不开放给第三方） | 直连抖音视频/直播/达人合作数据 |
| **货架场协同** | 部分通过抖店开放平台 | 全量原生 |
| **平台规则** | 需手动维护规则知识库 | 平台规则更新自动同步 |
| **触达入口** | 商家需主动打开 SaaS | 内嵌抖店主流程，触达成本为零 |
| **商家信任** | 需要授权敏感数据给第三方 | 平台官方，无信任成本 |
| **跨商家学习** | 数据私有，无法做行业基准 | 平台拥有全行业数据，可做匿名化基准建议 |

**核心结论**：抖店做这个产品，能利用的是**抖音电商生态数据 + 平台权威性 + 触达入口**——这是第三方 SaaS 永远做不到的。

## 3. 完整专家 Agent 矩阵（15 个）

按"服务消费者 → 新品冷启动 → 存量经营优化 → 内容场协同 → 决策辅助"五个层级组织：

### L1：服务消费者类（4 个）

| Agent | 核心能力 | 接入数据/工具 |
|---|---|---|
| **客服 Agent** | 消费者咨询自动应答（商品/退换货/优惠规则） | 商品库 / FAQ KB / 退换货政策 |
| **订单 Agent** | 订单查询/物流追踪/修改地址 | 订单 API / 物流 API |
| **售后 Agent** | 退款审批建议/纠纷预判/补偿方案推荐 | 售后规则库 / 历史纠纷案例 |
| **物流 Agent** | 发货提醒/物流异常预警/补发处理 | 物流接口 / 发货时效监控 |

### L2A：新品冷启动类（3 个）—— 服务工厂型 / 设计型 / 跨境型商家从 0→1

| Agent | 核心能力 | 关键能力点 |
|---|---|---|
| **新品立项 Agent** | 候选商品→S/A/B/C 等级评定→立项审批 | 按"年销额 / 日峰值 / 年净利润 / 毛利率"四指标自动定级；按"外观 / 品质 / 价格 / 品牌 / 视觉 / 营销 / 赛道增速"7 维加权竞争力评分；输出立项报告供商家钉钉/飞书审批 |
| **核价 Agent** | 原料+辅料+成品 SKU 组合 → 成本 / 售价 / 利润率核算 | 利润核算表自动生成；支持暂存 / 提交核价审批；采购价取最新通过记录；保留核价历史 |
| **上市节奏 Agent** | 自动排"卖点策划 / 拍摄 / 视觉制作 / 上架"时间轴 | 默认规则：交货前 15 天卖点策划、前 7 天拍摄、前 3 天视觉、入仓后 1 天上架；联动详情/主图/标题/SKU 设计 4 个营销环节自动建任务 |

> **为什么必须有这层**：抖店 30%+ 商家是工厂型/设计型，新品开发是核心刚需；存量优化只服务成熟商家，不解决新商家"0→1"上手问题。这层补齐后，Agent 真正覆盖"商品全生命周期"，而不只是"存量诊断"。

### L2B：存量经营优化类（5 个）—— 借鉴小龙虾思路最多的层

| Agent | 核心能力 | 灵感来源 |
|---|---|---|
| **选品 Agent** | AI 选品池/类目机会分析/价格带空窗发现 | 小龙虾"商机发现" |
| **商品诊断 Agent** | S/A 级大品每日异常诊断/近 7 天指标归因 | 小龙虾"商品诊断" |
| **关键词 Agent** | 30 天消费者需求趋势/差异化词根建议 | 小龙虾"关键词分析" |
| **竞品 Agent** | TOP 300 竞品自动对标/差异化策略建议 | 小龙虾"竞争格局分析" |
| **爆款策划 Agent** | 主图策划/爆款元素拆解（颜色/材质/价格） | 小龙虾"爆款策划" |

### L3：内容场协同类（2 个）—— 抖店独有

| Agent | 核心能力 | 抖店专属优势 |
|---|---|---|
| **直播策划 Agent** | 直播脚本生成/选品组合/GPM 预测/达人匹配 | 直连巨量百应数据 + 抖音直播 API |
| **短视频脚本 Agent** | 短视频带货脚本/爆款视频元素分析 | 抖音内容生态数据原生接入 |

### L4：决策辅助类（2 个）

| Agent | 核心能力 |
|---|---|
| **经营 OKR Agent** | 将商家年/月销售目标自动拆解为 OKR + 日任务，每日推送 |
| **罗盘归因 Agent** | 抖店罗盘指标异常自动归因 + 行动建议（"GMV 跌 12%，主因 SKU-A 停播 + 库存断货，建议补库存 + 加投千川 B 计划"） |

### Hub Agent（路由层）

负责理解商家自然语言意图，路由到对应专家。例如：
- "我今天该重点做什么" → 经营 OKR Agent + 罗盘归因 Agent 协同
- "帮我想想这个新品怎么打爆款" → 爆款策划 Agent + 竞品 Agent 串联
- "我有个新材质想立项" → 新品立项 Agent + 核价 Agent + 上市节奏 Agent 串联
- "这个客户问退货政策" → 客服 Agent + 售后 Agent

**Agent 间 Handoff 与 Skill 链**：每个专家 Agent 的 Skill MD 在 `next_skill` 字段声明下游推荐（如商品诊断 → 主图策划 / 关键词 → 爆款策划），形成可串联的 Skill 链而非孤岛。

## 4. 核心闭环：KPI → OKR → 日任务 → 复盘 → 沉淀

这是商家用本产品的**主操作主线**：

```
① 商家在抖店设定季度/年度销售目标
       ↓
② 经营 OKR Agent 自动拆解为月度 OKR（O + 3 个 KR）
       ↓
③ AI 拆解 KR 为日任务（带优先级、关联 SKU、关联负责人）
       ↓
④ 商家每日打开抖店首页 → 看到「专家方案 + 一键执行」
       ↓
⑤ AI 自动监控指标 / 日报周报月报生成
       ↓
⑥ 偏差时罗盘归因 Agent 自动诊断 + 建议调整
       ↓
⑦ 高质量结果回流到商家 Wiki，下次更准确
```

**关键设计**：
- 每个日任务带 **OKR 关联**（防止执行偏离目标）
- 每个日任务带 **SKU 关联**（自动继承 KR 的商品属性，防止操作错商品）
- 每次执行结果回写 → AI 下次更准

### 4.1 商家日常入口（操作界面四件套）

V2 不是后台工具，是**商家每天必看的经营驾驶舱**。打开抖店首页直接看到以下 4 个模块：

| 模块 | 功能 | 设计细节 |
|---|---|---|
| **月度进度看板** | 当月已完成/已逾期/整体达成进度条 | 支持月份切换，颜色区分进度区间，点击下钻到 KR 详情 |
| **当日工作建议** | AI 按当前进度自动生成 Top N 推荐工作 | 来自经营 OKR Agent + 罗盘归因 Agent 协同输出 |
| **AI 周期总结** | 日 / 周 / 月三档自动复盘报告 | 支持团队 / 个人视角切换；总结可一键同步到飞书多维表格或导出 PDF；含 KPI 达成度 + 异常归因 + 下周期 Top 行动建议 |
| **任务优先级** | 紧急 / 高 / 普通 / 低 四等级，颜色区分 | 拖动调整；AI 默认按"对 KR 贡献度 × 时间紧迫度"打分 |

### 4.2 OKR 入口（支持多种冷启动路径）

**新商家**怎么用上 Agent，是产品能否规模化的关键。提供 4 条 OKR 录入路径：

| 路径 | 说明 | 适用商家 |
|---|---|---|
| **AI 智能拆解** | 输入「商品 ID + 店铺名称 + 推广占比 + 销售金额」→ 系统关联抖店历史数据 → 自动生成具挑战性 OKR | 已有店铺、有历史数据 |
| **飞书多维表格同步** | 商家在多维表格里维护的目标表 → 一键同步进系统 | 已用飞书做经营管理的商家 |
| **Excel 导入** | 上传模板化 Excel → 系统识别 O 和 KR 结构 | 传统企业商家 |
| **手动新增** | 自由输入 O / KR → 非标场景兜底 | 新店、特殊品类 |

### 4.3 KR 协同执行

- **KR 团队指派**：KR 可指派给店铺运营 / 客服 / 主播等成员，系统实时同步完成状态与逾期预警
- **任务自动关联商品**：KR 中包含商品编号 / 链接 ID 时，系统自动建立关联，看板同步进度
- **任务继承商品属性**：日任务关联某 KR 时自动继承该 KR 绑定的商品类目 / 等级 / 价格带，专家 Agent 调用时无需重复传参

## 5. Skill MD 范式（举例：商品诊断专家）

```markdown
# Skill: 商品诊断

## 1. 角色定义
你是抖店 S/A 级大品的专属诊断专家，擅长基于罗盘近 7 天数据精准定位异常原因。

## 2. 核心目标
每日自动筛查 S/A 级商品的核心指标（CTR/CVR/GMV/UV 价值），定位异常并产出"创建日工作"的可执行建议。

## 3. SOP 标准作业流程
1. **数据拉取**：调用抖店罗盘 API，获取近 7 天 SKU 维度数据
2. **异常检测**:
   - CTR 环比下降 >15% → 主图异常
   - CVR 环比下降 >10% → 详情页/价格异常
   - UV 价值下降 >20% → 流量结构异常
3. **归因分析**：交叉对比类目均值、竞品数据、自身历史
4. **建议生成**：输出 3 条优先级排序的行动建议，每条带预估收益
5. **任务转化**：商家点击"创建日工作"，建议直接转日任务

## 4. 工具依赖
- `罗盘数据 API`：拉取 SKU 指标
- `类目均值查询`：对比基准
- `竞品快照查询`：横向参考
- `日任务创建工具`:写入抖店任务系统

## 5. 输入参数 (Schema)
- `seller_id`: 商家ID（必填）
- `product_ids`: 指定 SKU 列表，不填则全 S/A 级
- `time_window`: 默认 7 天

## 6. 输出规范
- `diagnosis`: 诊断结论 + 异常等级
- `root_cause`: 根因列表 + 证据引用
- `action_recommendations`: 行动建议（≤3 条）
- `next_skill`: 推荐联动技能（如"主图策划"）
```

**业务运营改 MD = 改产品策略**，不需要研发介入。这是 production-grade Agent 系统的标志。

### 5.1 工具协议层（MCP & Function Calling 合规）

Skill MD 之下还有一层关键工程：所有原子工具（罗盘 API / 千川 API / 物流接口 / 视觉生成 ...）必须符合行业标准协议，否则换模型 / 换 Agent 都得重写。

**三个工程要求**：

1. **JSON Schema 强制化**：每个工具定义完整的 input/output schema（字段类型、描述、required 约束），由协议适配层自动转换为 Claude / GPT / Gemini 各自的 function_call 格式 —— 同一份工具描述，跨模型可用
2. **MCP 兼容封装**：工具封装为 MCP Server 形式作为外部资源挂载，Agent 通过标准 `mcp_call_tool` 指令跨服务调用，**工具可在不同 Agent 系统间无缝迁移**（包括接入字节内部其他 Agent 平台）
3. **动态 Prompt 修正**：工具返回不符合 schema 时，技能引擎根据 MCP 反馈协议**自动生成补救 Prompt** 引导模型修正参数，而不是让 Agent 卡死

**工具自动注册**：新开发的工具只需继承 `BaseTool` 类并完善注解，系统启动时**自动解析生成 OpenAI / Anthropic 格式的工具库快照**，无需手动注册。工具元数据与 Skill MD 实时校验，确保 SOP 每一步执行都是类型安全的。

**为什么这层很重要**：
- 抖店 Agent 矩阵未来要接入字节其他 AI 平台（豆包 / Coze / 火山引擎方舟），协议合规是前置条件
- 工具数量超 200 后，类型安全是稳定性底线
- 行业标准协议让"换底层大模型"成本降到接近 0

> **借鉴依据**：参考 Anthropic 的 MCP 标准与 OpenAI Function Calling 规范，对应工程范式（小龙虾团队 UniversalSkillEngine 思路）。

## 6. 数据底座（抖音生态原生）

| 数据源 | 提供能力 | 接入方式 |
|---|---|---|
| **抖店罗盘** | 实时 GMV/CTR/CVR/转化漏斗 | 内部 API 直连 |
| **抖音内容数据** | 短视频/直播/达人合作流量数据 | 抖音内部数据中台 |
| **巨量千川** | 投流计划/ROI/人群圈选 | 千川 API |
| **巨量百应** | 达人撮合/直播商品池 | 百应数据 |
| **平台规则库** | 实时同步抖店规则更新 | 内部知识系统 |
| **视觉证据库** | 商品主图 / 短视频帧 / 直播切片的视觉资产索引（VLM caption + 标签 + 热区） | VLM 批量解析 + 索引服务 |
| **商家私域** | 商家上传的 SKU 详情/客服话术/历史经营 | 商家自助导入 |
| **匿名行业基准** | 同类目商家匿名化数据聚合 | 平台数据脱敏后产出 |

**第三方 SaaS 拿不到的**：抖音内容数据 + 百应达人数据 + 实时平台规则 + 视觉证据库 + 匿名行业基准。

### 6.1 视觉证据为什么是一等公民

抖音电商是视觉主导的电商——主图点击率、短视频完播率、直播切片转化率，几乎所有关键指标都和视觉强相关。视觉证据不能只当成 Markdown 附件，要作为**可检索、可引用、可比对**的一等资产：

| 资产类型 | 存储形态 | 用途 |
|---|---|---|
| 主图证据 | image-index.json（imageId + caption + 标签 + 类目 + GMV） | 爆款策划 Agent 找参考；竞品 Agent 横向对比 |
| 短视频帧 | 关键帧抽取 + VLM caption | 短视频脚本 Agent 拆解爆款元素 |
| 直播切片 | 时间戳 + ASR + 画面 caption | 直播策划 Agent 复盘高 GPM 时段 |
| 商家自有素材 | 商家私有 overlay | 视觉风格一致性约束 |

**问答时优先视觉证据召回**：当商家问"我这张主图为什么 CTR 低"，Agent 不应纯文字回答，而要**取出问题主图 + 同品类爆款主图，并排展示**，证据卡独立渲染。这是抖音电商场景下"AI 可解释性"最重要的一环。

## 9. 风险与边界（5 道防护线）

1. **数据边界**：商家私有 Wiki 严格租户隔离，不跨商家泄露
2. **事实边界**：AI 回答必须有 evidence ref（引用某条数据/某个规则），不允许"凭空生成"
3. **自动化边界**：AI 默认只输出可审阅产物（建议），不直接执行高风险动作（改价/退款/上下架）
4. **成本边界**：高 token 操作（深度研究/视觉识别）显式标价 + 限流，月度成本透明
5. **合规边界**：所有 Agent 输出都符合广告法/电商法/平台规则（行业 Wiki 内置合规知识）

### 9.1 统一证据引用接口（事实边界的工程落地）

"必须有 evidence ref"不是一句话，是一份强制约束的接口契约。每条 AI 输出都必须挂上一个或多个 KnowledgeEvidenceRef：

```ts
interface KnowledgeEvidenceRef {
  refId: string
  layer: "raw" | "wiki" | "image" | "semantic_unit" | "strategy_action" | "skill_run"
  // 来源 6 选 1：原始文档 / Wiki 页 / 视觉证据 / 语义单元 / 策略动作卡 / Skill 执行产物
  sourceId?: string       // 行业 Wiki 来源
  pageId?: string         // 商家 Wiki 页
  imageId?: string        // 视觉证据
  semanticUnitId?: string // 语义规则
  quoteOrSummary: string  // 引用原文或摘要
  confidence: number      // 0–1
  status: "candidate" | "confirmed" | "rejected" | "superseded"
}
```

**好处**：
- 商家在 AI 建议旁边点"查看依据" → 直接跳转到行业 Wiki 原文 / 商家自己上传的 SOP / 罗盘原始指标 / 主图对比卡
- 当 Wiki 更新或冲突裁决后，所有引用过该证据的历史 Agent 输出可批量标记为 `superseded`，避免商家用过时建议
- 跨 6 个 layer 统一接口，未来扩展（视频证据 / 直播切片）只需新增 layer 枚举

### 9.2 Agent 运行可观测性

每次 Agent 调用记录完整执行 timeline（输入校验 / 上下文读取 / 执行计划 / 工具调用 / 降级原因 / 成本），商家可在"AI 助手运行历史"里查看：
- 这次回答用了哪些数据 / 哪些 Wiki 页 / 哪些工具
- 执行计划是什么、为什么走了某条路径
- 消耗了多少 token、花费多少钱
- 如果触发降级 / 错误，原因是什么

> 这是"商家信任 AI"的最后一道地基：**不可观测的 Agent = 黑盒 = 不敢用**。

## 10. 高频面试追问（10 个）速答

### Q1：为什么是 Multi-Agent 不是单一 Agent？
> 工具数量 200+ 超出单 Agent 工具上限；商家场景非线性（用户随时跳转上下文）；专业领域差异大（选品逻辑 ≠ 客服话术）。Wardrobe 项目验证 Handoff 模式准确率 95%+。

### Q2：Skill MD 范式相比直接写代码的优势？
> 业务运营专家可改 MD 优化策略，不需要研发介入。OpenClude/Anthropic Skills 验证过的工程范式。代码与策略解耦，迭代速度提升 10 倍。

### Q3：为什么是双层 Wiki 不是单一知识库？
> 行业 Wiki 是平台预编译的共性知识（指标口径/平台规则/类目 SOP），所有商家共享；商家 Wiki 是私有 overlay（自有 SKU/话术/历史），不外泄。冲突按 6 种语义关系（supports / refines / scope_differs / contradicts / supersedes / duplicates）分别处理，不是简单覆盖。这避免了"千店一面"和"知识无法复用"两个极端。

### Q4：如何避免 AI 答错？
> 六层防护：① RAG 强约束（必须 KnowledgeEvidenceRef 引用，6 个 layer 任选） ② Skill MD 限定 SOP ③ 输出 JSON Schema 校验 ④ 视觉证据优先召回（抖音电商视觉主导，文字回答 + 图片证据卡双轨） ⑤ 高风险动作强制 Review ⑥ 错误回复自动回流 ReviewItem 改进 Skill。

### Q5：商家凭什么信任 AI 的诊断和建议？
> 四个层次：① 平台权威（不是第三方）② 可解释性（每条建议附完整 KnowledgeEvidenceRef，可点击跳转原文/图片证据/原始指标）③ 可观测性（Agent 运行历史里能看完整 timeline + 成本 + 降级原因）④ 灰度验证（先 1% 商家，效果验证后扩量）。

### Q6：和千牛 / 巨量百应的差异化？
> 千牛是单 Agent + 接大模型，能力浅；百应是直播单场景。我们是**商家全生命周期闭环（新品冷启动 + 存量经营 + 内容场 + 决策辅助）+ 双层 Wiki + 15 专家矩阵 + MCP 协议合规 + 抖音生态原生数据 + 视觉证据一等公民**。最关键差异：内容场 + 货架场协同（千牛只有货架，百应只有内容），且覆盖新品从 0→1（千牛仅服务存量）。

### Q7：成本怎么算？
> 单商家日均 50 次 Agent 调用 × 0.05 元/次 = 75 元/月。商家平均节省人力 1.2 万元/月。ROI > 100 倍。平台侧成本线性扩张，但商家粘性增长带来的 GMV 增长是平方级。

### Q8：如何衡量 Agent 是否成功？
> 北极星：月活商家 × Agent 使用率 × 月均节省时长。漏斗：开启率 30% / 日均调用 10 次 / 任务完成率 75% / GMV 增量 +8% / D30 留存 60%。对立指标：错误申诉率 / CSAT / 误操作率防单边优化。

### Q9：行业 Wiki 怎么维护？谁来更新？
> 平台知识工程团队负责。每月发布 version release，配套行业 eval cases（评测集）做发布前验收。商家可提交匿名化反馈进入"行业知识共创"路径，但需平台审稿。

### Q10：15 个 Agent 都做完要多久？MVP 怎么选？
> MVP 选最高 ROI 三个：客服 Agent（覆盖 70% 商家咨询场景）+ 商品诊断 Agent（替代抖店罗盘的低效看数据流程）+ 经营 OKR Agent（拉新商家上手）。4 周内可上线灰度。完整 15 个 Agent 12 个月内完成，按"存量经营 → 内容场 → 新品冷启动 → 决策辅助"四波次推进。新品冷启动放后期是因为依赖核价 / 上市排程等线下流程接口，不依赖抖音侧实时数据，可与抖店现有 ERP 集成方案并行筹备。
