## 1. 项目是什么（30 秒看懂）

**这是抖店 AI 助手的"中小商家专版"**，专门服务平台 90% 拿不到人工小二的 S4/S5 店家。

它覆盖商家全生命周期的**两条业务线**：

| 业务线 | 用户阶段 | 子图 | 负责人 |
|---|---|---|---|
| **冷启动选品** | 新商家入驻、第一次决定卖什么 | 商机洞察子图 | 甘华梁 |
| **在运营诊断** | 已有数据但拿不到小二、看不懂罗盘 | 经营诊断子图 | 贾丽婼 |

两条线**共用同一套 Agent 框架**（基座 + 可插拔 Skill MD + 双层 Wiki + Hub 路由）。换品类只换 MD，换分析任务只加子图——这是平台，不是单一工具。

---

## 2. 为什么做这个（产品起点）

### 2.1 平台事实：中小商家被结构性放弃

- 1 个类目运营小二要对接 **300-800 家**中小商家，60% 工时被头部 KA 占用
- **90% 平稳经营、数据偏弱的中小商家全年不会收到 1 次小二主动私聊**
- 不是小二不努力——是平台 GMV 锚定决定的制度结果，**不可能配人工一对一帮扶**

平台已经把这 90% 交给 AI，但**接管质量很烂**。

### 2.2 现有抖店 AI 助手的 4 大短板

| 短板 | 表现 | 商家感受 |
|---|---|---|
| ① 报问题不找根因 | "转化率下跌，建议优化详情" | 我知道跌了，我想知道为什么跌 |
| ② 同质化套话 | 服饰/食品/家居共用一套话术；自播店和货架店给一样建议 | AI 根本不懂我做的是什么 |
| ③ 缺资源配套 | "去报活动拉流量" | 报哪个？我够格吗？怎么才够格？ |
| ④ 不处理非数据信息 | 商城精选频道上线、活动门槛变化，AI 完全不知道 | 我数据没问题但流量莫名其妙跌了 |

### 2.3 新商家入驻同时存在另一个困境

新商家第一次选品时，没有历史数据可看、没有同行参照，靠经验拍脑袋——这是甘华梁子图要解的问题。

**两个困境合起来覆盖了商家全生命周期**：「我该卖什么」（选品）+ 「我已经在卖了，但不知道哪里出问题」（诊断）。

## 3. 技术基座（四层架构）

整套系统不是两个孤立工具，而是建在一个**共用基座**上，两个子图独立编排，通过 Hub 按意图路由。

```
┌─────────────────────────────────────────────────┐
│                  Hub 路由                         │
│  （按用户自然语言意图分发到对应子图）              │
└────────────┬───────────────────────┬────────────┘
             ↓                       ↓
   ┌──────────────────┐    ┌──────────────────┐
   │  商机洞察子图     │    │   经营诊断子图    │
   │ （冷启动选品）    │    │ （在运营商家）    │
   │     9 节点        │    │     4 节点        │
   └──────────┬───────┘    └─────────┬────────┘
              ↓                      ↓
   ┌──────────────────────────────────────────┐
   │              共用基座                      │
   │  state · evidence · review · wiki ·       │
   │  tools 协议 · skills loader · LangGraph    │
   └──────────────────────────────────────────┘
```

### 3.1 基座（agent/）—— 两条业务线绝对不许独自改

| 模块 | 文件 | 职责 |
|---|---|---|
| **状态层** | `agent/state.py` | `InsightState`（选品）/ `DiagnosisState`（诊断）独立定义，TypedDict 增量合并 |
| **证据引用** | `agent/evidence.py` | 每条 AI 结论挂 `KnowledgeEvidenceRef`（layer + sourceId + quote + confidence），空结论不让出 |
| **人审状态机** | `agent/review.py` | 高风险动作（改价 / 改活动）走候选 → 人工确认 → 执行 |
| **双层 Wiki** | `agent/wiki/{industry,seller}.py` | 行业 Wiki 共用只读；商家 Wiki 按店 overlay 个性化 |
| **工具协议** | `agent/tools/base.py` | `BaseTool` + `TOOL_REGISTRY`，统一 JSON Schema |
| **Skill 加载器** | `agent/skills/loader.py` | 把 MD 文件解析成可调用的专家 |
| **图后端** | `agent/graph.py` | MiniGraph（零依赖兜底）↔ LangGraph 一键切换 |
| **路由入口** | `agent/hub.py` | 按意图分发；扩展新子图加一行 |
| **CLI 入口** | `agent/run.py` | 统一命令行，两条线共用 |

### 3.2 Skill MD（agent/skills/）—— 配置即知识

每个"专家"和每个"场景"都是一份 markdown：

- 业务侧能改（产品/运营/老小二），不用碰代码
- 新加专家 = 新增一份 MD
- 新加品类 = 新增一组 overlay MD（如 `nanzhuang_*.md`），加载器零改动

**统一格式**（6 章节，loader 按 regex 解析）：
```
# Skill: 名字
## 1. 角色定义
## 2. 核心目标
## 3. SOP（每步必带决策点 + 工具 + fallback）
## 4. 工具依赖
## 5. 输入 schema
## 6. 输出规范（含 evidence_refs / confidence / next_skill）
```

### 3.3 双层 Wiki（agent/wiki/）—— 解决"AI 不处理非数据信号"短板

| 层 | 内容 | 共享性 |
|---|---|---|
| **行业 Wiki** | 类目基线 / 活动库 / 工具库 / 规则变动 | 全平台共用只读 |
| **商家 Wiki** | 单店 overlay（这家店上次诊断结论 / 已采纳建议 / 历史改善记录） | 按店一份，跨子图引用 |

行业 Wiki 当前 4 份 MD：
- `category_baseline.md` —— 女装 6 个细分场景的健康度基线
- `activity_db.md` —— 10 个女装相关平台活动（含准入门槛 / 历史命中率）
- `tool_db.md` —— 23 个官方工具入口（AIGC / SEO / 千川 / 达人 / 商城 / 话术）
- `rule_changes.md` —— 5 条近 30 天规则变动 + 2 条预告

### 3.4 LLM / 数据 / 部署假设

- **LLM**：OpenAI SDK 协议，默认 DashScope 千问；改 `.env LLM_BASE_URL` 即可切 DeepSeek / 智谱 / Kimi，**不换包**
- **数据**：选品走 TikHub API；诊断走抖店罗盘字段（V1 用 mock JSON 还原 5 个 case）
- **运行**：Python 3.12+，`AGENT_MOCK=1` 离线全链路跑通，无 key 不崩

## 4. 业务模块 A · 冷启动选品（甘华梁）

### 4.1 解决什么

新商家入驻、第一次选品时，**没有历史数据可看、没有差异化方向、不知道竞品在做什么**。

### 4.2 子图结构（9 节点）

```
KeywordPlanner      （拓出 5-15 个精准关键词）
       ↓
┌─── SocialAnalyzer    （人群洞察）
├─── PriceAnalyzer     （价格带 + 商品池打分）
├─── HotItemAnalyzer   （爆款共性提炼）
└─── CompetitorAnalyzer（竞品痛点扫描）
       ↓ 并行
   Collector          （汇总 raw → curated）
       ↓
   Curator            （去重排序）
       ↓
   Briefing           （分章节简报）
       ↓
   Editor             （生成 markdown 报告）
```

### 4.3 专家 Skill MD（5 个）

`关键词` / `选品` / `竞品` / `爆款` / `口碑诊断`

### 4.4 输入输出

```bash
python3 -m agent.run "口红" --category 美妆 --topn 20 --mock
```

输出：Markdown 商机洞察报告，含 5 章节（人群 / 价格 / 爆款 / 竞品 / 风险）+ 57 条证据引用。

### 4.5 数据源

TikHub API（外部）—— 通过 `agent/tools/tikhub_tools.py` 封装成 6 个标准工具：`search_products` / `rank_products` / `fetch_reviews` / `analyze_reviews` / `snapshot_write` / `trend_query`。

## 5. 业务模块 B · 在运营诊断（贾丽婼）

### 5.1 解决什么

抖店 S4/S5 中小商家——**已经在卖了**，但拿不到人工小二、看不懂罗盘、AI 助手只会空话。本子图直接补齐前述 4 大短板。

### 5.2 子图结构（4 节点）

```
Checker     （体检 · 异常识别）
   │  调用：fetch_shop_metrics + category_baseline
   │  产出：anomalies[] + matched_overlays[] + data_completeness
   ↓
Attributor  （归因 · MECE 拆维度 + 非数据信号检索）
   │  调用：LLM + 类目 overlay SOP + rule_changes Wiki
   │  产出：root_cause_chains[] + non_data_signals[]
   ↓
Advisor     （建议 · 资源挂钩 + 准入校验）
   │  调用：activity_db + tool_db Wiki + LLM
   │  产出：actions[]（每条挂 resource.url + eligibility）
   ↓
Composer    （组装 · Editor 风格 markdown）
   │  产出：完整体检报告
   ↓
   商家 / 小二端
```

## 6. 输出形态

### 6.1 商家端 · 体检报告（Markdown 卡片）

嵌入抖店 AI 助手对话框 / 商家首页推送：

```markdown
# 经营诊断报告 · 咖喱生活优品铺子

> 类目：女装>连衣裙 · 等级：L3 · 体验分：4.3

## 核心问题
**千川投产比显著下跌（55.6%）**

> 适用画像：女装成长期 / 女装自播店 · 数据完整度 100%

## 异常指标清单
- **千川投产比** 当前 0.8 / 基线 1.8 / 偏离 -55.6%
- **访客价值** 当前 1.46 / 基线 10.0 / 偏离 -85.4%
- ……

## 为什么会这样（根因链）
### 异常指标：千川投产比
- 千川投产比（0.8）显著偏离类目基线（1.8）· 置信度 0.92
- 转化率同期异动，存在相关性 · 置信度 0.75

## 非数据信号
- 商城精选频道升级（2026-05-28），未入驻商家搜索分流 15-25%

## 本周建议（按性价比 × 置信度排序）
1. 女装连衣裙千川计划模板 · 性价比高 · 置信度 0.92
2. 商城精选入驻申请 · 性价比中 · 准入达成
3. ……
```

### 6.2 小二端 · 潜力候选名单（周维度，间接价值）

每周自动生成「AI 已诊断 + 商家已采纳 + 指标已回升」的店推送给小二：

```markdown
## S5 → S3 潜力候选（本周 12 家）
| 店铺 | 已采纳建议 | 14 天指标改善 |
|---|---|---|
| 咖喱生活 | 千川计划模板 + 主图重做 | CVR 1.3% → 1.9% |
| ……
```

这部分是「平台战略价值」——为小二的 S3 潜力池供高质量货源，从根上改善小二人力效率。

### 6.3 演示界面 · Streamlit demo

`demo/streamlit_app.py`：左侧选 case → 点"生成报告" → 1 秒出完整诊断 + 6 个下钻面板（异常指标可视化 / 根因链 / 非数据信号 / 行动建议 / Wiki 命中条目 / 节点执行轨迹）。

```bash
pip install -r requirements.txt
streamlit run demo/streamlit_app.py
# http://localhost:8501
```

实时调 `agent.graph.run_diagnosis`，mock 模式无需任何 API key。

### 6.4 输出契约（所有出口共享）

无论商家端 / 小二端 / demo，所有 AI 结论都满足三条硬规则：

1. **挂证据**：每条结论必须有 `KnowledgeEvidenceRef`（来源层 + 原文 quote + 置信度）
2. **挂资源**：每条行动建议必须有 `resource.url`（活动 ID / 工具 / 模板 / 任务）
3. **挂准入**：建议的资源若有门槛（如体验分 ≥ 4.5），必须标注当前是否达成 + 缺口

## 7. 复用性 · 为什么这是平台不是工具

这是项目最值钱的设计点。**底层框架在 V1 就被同时跑过两个完全不同的子图验证**（甘华梁的选品 9 节点 + 贾丽婼的诊断 4 节点），意味着扩展边际成本是已知的。

### 7.1 三个维度的可扩展

| 想扩什么 | 改什么 | 不改什么 |
|---|---|---|
| **新品类**（男装 / 食品 / 家居） | 新增一组 overlay MD（如 `nanzhuang_*.md`） | 代码、graph、hub、所有节点 |
| **新分析任务**（趋势复盘 / 选品聚类 / 潜力商家挖掘） | 新增一个子图 `agent/nodes/<task>/*.py` + Hub 加一条意图分支 | 基座、其他子图 |
| **新 LLM 提供商** | 改 `.env LLM_BASE_URL` | 一行代码不动 |
| **新图后端** | `AGENT_USE_LANGGRAPH=1` | 子图代码不动（接口一致） |

### 7.2 已经验证的扩展点

- ✅ **不同子图共底座**（甘华梁选品 + 贾丽婼诊断 = 2 个独立子图，state / wiki / evidence 接口完全复用）
- ✅ **跨子图路由**（女装成长期 overlay 在诊断时可路由到甘华梁的选品专家做"新品引流"建议）
- ✅ **MiniGraph ↔ LangGraph 切换**（环境变量 `AGENT_USE_LANGGRAPH=1` 切，子图代码不动）

### 7.3 未来可扩的方向（不是承诺，是说明边际）

| 方向 | 改动范围 | 时间估 |
|---|---|---|
| 男装 / 童装 / 食品 V1 | 写一套 overlay MD + 类目基线 Wiki | 1-2 周 |
| 选品趋势聚类（抓 10w 数据做用户聚类） | 加新子图（聚类节点） + Hub 加意图分支 | 2-3 周 |
| 接入抖店 AI 助手真实对话框 | 主要是工程接入 + 推送策略，框架不动 | 跨团队协同 |
| 小二端潜力池推送 | 加一个调度任务读 evidence_refs + Review 状态机 | 1-2 周 |

## 10. 一句话总结

> 90% 的中小商家拿不到小二，被 AI 接管了但接管质量很烂。
>
> 本项目不是再做一个"小二副驾"——小二的 AI 排序已经够好——而是**直接补齐 AI 接管中小商家时的具体短板**：不找根因 → 加归因子图；同质化 → 类目 overlay；缺资源 → 资源数据库挂钩；不处理规则 → 双层 Wiki。
>
> 技术上是一套**可插拔的 Agent 平台**：共用基座 + Skill MD + 双层 Wiki + Hub 路由。V1 同时跑通选品（甘华梁）+ 诊断（贾丽婼）两个子图——这已经证明换品类只换 MD、换分析任务只加子图，扩展边际成本是已知的。

跑通的不是一个工具，是一种能持续扩张的产品形态。
