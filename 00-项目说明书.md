## 1. 项目是什么（30 秒看懂）

**这是抖店 AI 助手的"中小商家专版"**，专门服务平台 90% 拿不到人工小二的 S4/S5 店家。

它覆盖商家全生命周期的**两条业务线**：

| 业务线 | 用户阶段 | 子图 | 负责人 |
|---|---|---|---|
| **冷启动选品** | 新商家入驻、第一次决定卖什么 | 商机洞察子图 | 甘华梁 |
| **在运营诊断** | 已有数据但拿不到小二、看不懂罗盘 | 经营诊断子图 | 贾丽婼 |

两条线**共用同一套 Agent 框架**（基座 + 可插拔 Skill MD + 双层 Wiki + Hub 路由）。换品类只换 MD，换分析任务只加子图——这是平台，不是单一工具。

---

## 2. 为什么做这个（产品起点）

### 2.1 平台事实：中小商家被结构性放弃

- 1 个类目运营小二要对接 **300-800 家**中小商家，60% 工时被头部 KA 占用
- **90% 平稳经营、数据偏弱的中小商家全年不会收到 1 次小二主动私聊**
- 不是小二不努力——是平台 GMV 锚定决定的制度结果，**不可能配人工一对一帮扶**

平台已经把这 90% 交给 AI，但**接管质量很烂**。

### 2.2 现有抖店 AI 助手的 4 大短板

| 短板 | 表现 | 商家感受 |
|---|---|---|
| ① 报问题不找根因 | "转化率下跌，建议优化详情" | 我知道跌了，我想知道为什么跌 |
| ② 同质化套话 | 服饰/食品/家居共用一套话术；自播店和货架店给一样建议 | AI 根本不懂我做的是什么 |
| ③ 缺资源配套 | "去报活动拉流量" | 报哪个？我够格吗？怎么才够格？ |
| ④ 不处理非数据信息 | 商城精选频道上线、活动门槛变化，AI 完全不知道 | 我数据没问题但流量莫名其妙跌了 |

### 2.3 新商家入驻同时存在另一个困境

新商家第一次选品时，没有历史数据可看、没有同行参照，靠经验拍脑袋——这是甘华梁子图要解的问题。

**两个困境合起来覆盖了商家全生命周期**：「我该卖什么」（选品）+ 「我已经在卖了，但不知道哪里出问题」（诊断）。

## 3. 技术基座（四层架构）

整套系统不是两个孤立工具，而是建在一个**共用基座**上，两个子图独立编排，通过 Hub 按意图路由。

```
┌─────────────────────────────────────────────────┐
│                  Hub 路由                         │
│  （按用户自然语言意图分发到对应子图）              │
└────────────┬───────────────────────┬────────────┘
             ↓                       ↓
   ┌──────────────────┐    ┌──────────────────┐
   │  商机洞察子图     │    │   经营诊断子图    │
   │ （冷启动选品）    │    │ （在运营商家）    │
   │     9 节点        │    │     4 节点        │
   └──────────┬───────┘    └─────────┬────────┘
              ↓                      ↓
   ┌──────────────────────────────────────────┐
   │              共用基座                      │
   │  state · evidence · review · wiki ·       │
   │  tools 协议 · skills loader · LangGraph    │
   └──────────────────────────────────────────┘
```

### 3.1 基座（agent/）—— 两条业务线绝对不许独自改

| 模块 | 文件 | 职责 |
|---|---|---|
| **状态层** | `agent/state.py` | `InsightState`（选品）/ `DiagnosisState`（诊断）独立定义，TypedDict 增量合并 |
| **证据引用** | `agent/evidence.py` | 每条 AI 结论挂 `KnowledgeEvidenceRef`（layer + sourceId + quote + confidence），空结论不让出 |
| **人审状态机** | `agent/review.py` | 高风险动作（改价 / 改活动）走候选 → 人工确认 → 执行 |
| **双层 Wiki** | `agent/wiki/{industry,seller}.py` | 行业 Wiki 共用只读；商家 Wiki 按店 overlay 个性化 |
| **工具协议** | `agent/tools/base.py` | `BaseTool` + `TOOL_REGISTRY`，统一 JSON Schema |
| **Skill 加载器** | `agent/skills/loader.py` | 把 MD 文件解析成可调用的专家 |
| **图后端** | `agent/graph.py` | MiniGraph（零依赖兜底）↔ LangGraph 一键切换 |
| **路由入口** | `agent/hub.py` | 按意图分发；扩展新子图加一行 |
| **CLI 入口** | `agent/run.py` | 统一命令行，两条线共用 |

### 3.2 Skill MD（agent/skills/）—— 配置即知识

每个"专家"和每个"场景"都是一份 markdown：

- 业务侧能改（产品/运营/老小二），不用碰代码
- 新加专家 = 新增一份 MD
- 新加品类 = 新增一组 overlay MD（如 `nanzhuang_*.md`），加载器零改动

**统一格式**（6 章节，loader 按 regex 解析）：
```
# Skill: 名字
## 1. 角色定义
## 2. 核心目标
## 3. SOP（每步必带决策点 + 工具 + fallback）
## 4. 工具依赖
## 5. 输入 schema
## 6. 输出规范（含 evidence_refs / confidence / next_skill）
```

### 3.3 双层 Wiki（agent/wiki/）—— 解决"AI 不处理非数据信号"短板

| 层 | 内容 | 共享性 |
|---|---|---|
| **行业 Wiki** | 类目基线 / 活动库 / 工具库 / 规则变动 | 全平台共用只读 |
| **商家 Wiki** | 单店 overlay（这家店上次诊断结论 / 已采纳建议 / 历史改善记录） | 按店一份，跨子图引用 |

行业 Wiki 当前 4 份 MD：
- `category_baseline.md` —— 女装 6 个细分场景的健康度基线
- `activity_db.md` —— 10 个女装相关平台活动（含准入门槛 / 历史命中率）
- `tool_db.md` —— 23 个官方工具入口（AIGC / SEO / 千川 / 达人 / 商城 / 话术）
- `rule_changes.md` —— 5 条近 30 天规则变动 + 2 条预告

### 3.4 LLM / 数据 / 部署假设

- **LLM**：OpenAI SDK 协议，默认 DashScope 千问；改 `.env LLM_BASE_URL` 即可切 DeepSeek / 智谱 / Kimi，**不换包**
- **数据**：选品走 TikHub API；诊断走抖店罗盘字段（V1 用 mock JSON 还原 5 个 case）
- **运行**：Python 3.12+，`AGENT_MOCK=1` 离线全链路跑通，无 key 不崩

## 4. 业务模块 A · 冷启动选品（甘华梁）

### 4.1 解决什么

新商家入驻、第一次选品时，**没有历史数据可看、没有差异化方向、不知道竞品在做什么**。

### 4.2 子图结构（9 节点）

```
KeywordPlanner      （拓出 5-15 个精准关键词）
       ↓
┌─── SocialAnalyzer    （人群洞察）
├─── PriceAnalyzer     （价格带 + 商品池打分）
├─── HotItemAnalyzer   （爆款共性提炼）
└─── CompetitorAnalyzer（竞品痛点扫描）
       ↓ 并行
   Collector          （汇总 raw → curated）
       ↓
   Curator            （去重排序）
       ↓
   Briefing           （分章节简报）
       ↓
   Editor             （生成 markdown 报告）
```

### 4.3 专家 Skill MD（5 个）

`关键词` / `选品` / `竞品` / `爆款` / `口碑诊断`

### 4.4 输入输出

```bash
python3 -m agent.run "口红" --category 美妆 --topn 20 --mock
```

输出：Markdown 商机洞察报告，含 5 章节（人群 / 价格 / 爆款 / 竞品 / 风险）+ 57 条证据引用。

### 4.5 数据源

TikHub API（外部）—— 通过 `agent/tools/tikhub_tools.py` 封装成 6 个标准工具：`search_products` / `rank_products` / `fetch_reviews` / `analyze_reviews` / `snapshot_write` / `trend_query`。

## 5. 业务模块 B · 在运营诊断（贾丽婼）

### 5.1 解决什么

抖店 S4/S5 中小商家——**已经在卖了**，但拿不到人工小二、看不懂罗盘、AI 助手只会空话。本子图直接补齐前述 4 大短板。

### 5.2 子图结构（4 节点）

```
Checker     （体检 · 异常识别）
   │  调用：fetch_shop_metrics + category_baseline
   │  产出：anomalies[] + matched_overlays[] + data_completeness
   ↓
Attributor  （归因 · MECE 拆维度 + 非数据信号检索）
   │  调用：LLM + 类目 overlay SOP + rule_changes Wiki

   │  产出：actions[]（每条挂 resource.url + eligibility）
   ↓
Composer    （组装 · Editor 风格 markdown）
   │  产出：完整体检报告
   ↓
   商家 / 小二端
```

## 6. 输出形态

### 6.1 商家端 · 体检报告（老板视角 · 人话叙事）

嵌入抖店 AI 助手对话框 / 商家首页推送。**S4/S5 老板看到的是这个样子**——没有工程指标，没有置信度，没有根因链结构，只有人话 + 三件事 + 一件提醒：

```markdown
# 你的店本周诊断 · 咖喱生活

近 7 天卖了 12 万，比上周少了 16 万——差不多砍半。

跌得最狠的是直播：进直播间的人只待 22 秒就走，同行能留 1 分钟以上。
千川也在亏——你花 100 元投放，从原来能换 160 元成交，跌到只换 80 元。

## 这周必须做的三件事

**1. 直播间话术换一版 · 今晚就能换 · 不花钱**
- 用平台的话术模板（→ 打开），福袋要开播 5 分钟内发，别等到中场
- 预计停留能从 22 秒拉到 40 秒以上

**2. 千川计划全停 + 按平台模板重启 · 后台 10 分钟**
- 你现在 4 个计划都在亏（每 100 元投回 80 元），继续投是赔本
- 平台有一套女装连衣裙的千川模板（→ 打开），照这个建新计划
- 预计 3 天看到效果

**3. 申请「商城精选」频道 · 你够资格但还没申请**
- 平台上月新出的流量入口，你评价分 4.6 够准入
- 没申请的店搜索流量被分走 15-25%——你这两周流量跌就有这个原因
- 申请流程 1 周入选

## 一件事提醒你

体验分还差 0.2 就到 4.5 的活动准入线。下周如果冲上去，618 女装预热活动就能报名，估算多带 20w 左右成交。

最快的提分办法：把这 3 个高退款 SKU 先下架（→ 查看名单）。
```

**为什么改成这样**

| 商家端不出现 | 但 Agent 内部 / 小二 / 团队版仍保留 |
|---|---|
| `千川投产比` / `访客价值` / `转粉率` 等工程 metric | ✓ 保留在 `DiagnosisState.anomalies` |
| 置信度 / 严重度 / 性价比 / 数据完整度 | ✓ 保留在 advisor / checker 节点输出 |
| 根因链结构（`primary_root_cause_index`） | ✓ 保留在 `root_cause_chains` |
| Wiki 命中 / 节点 trace / evidence_refs | ✓ 保留在 final_state，给小二/调试用 |

所有结构化数据**都还在 Agent 内部**，只是商家视图层做了「人话翻译 + 老板心智重组」。同一份 state 可输出三种视图（详见 §9.1）：

- `view=owner`：本节示例（老板版，1 句问题 + 3 条按钮 + 1 条提醒）
- `view=ops`：当前 Streamlit demo 形态（漏斗诊断 + 根因链 + 建议表）
- `view=team`：完整指标 + SKU 维度 + 投流明细（给有专职运营的店）

### 6.4 输出契约（所有出口共享）

无论商家端 / 小二端 / demo，所有 AI 结论都满足三条硬规则：

1. **挂证据**：每条结论必须有 `KnowledgeEvidenceRef`（来源层 + 原文 quote + 置信度）
2. **挂资源**：每条行动建议必须有 `resource.url`（活动 ID / 工具 / 模板 / 任务）
3. **挂准入**：建议的资源若有门槛（如体验分 ≥ 4.5），必须标注当前是否达成 + 缺口

## 7. 复用性 · 为什么这是平台不是工具

这是项目最值钱的设计点。**底层框架在 V1 就被同时跑过两个完全不同的子图验证**（甘华梁的选品 9 节点 + 贾丽婼的诊断 4 节点），意味着扩展边际成本是已知的。

### 7.1 三个维度的可扩展

| 想扩什么 | 改什么 | 不改什么 |
|---|---|---|
| **新品类**（男装 / 食品 / 家居） | 新增一组 overlay MD（如 `nanzhuang_*.md`） | 代码、graph、hub、所有节点 |
| **新分析任务**（趋势复盘 / 选品聚类 / 潜力商家挖掘） | 新增一个子图 `agent/nodes/<task>/*.py` + Hub 加一条意图分支 | 基座、其他子图 |
| **新 LLM 提供商** | 改 `.env LLM_BASE_URL` | 一行代码不动 |
| **新图后端** | `AGENT_USE_LANGGRAPH=1` | 子图代码不动（接口一致） |

### 7.2 已经验证的扩展点

- ✅ **不同子图共底座**（甘华梁选品 + 贾丽婼诊断 = 2 个独立子图，state / wiki / evidence 接口完全复用）
- ✅ **跨子图路由**（女装成长期 overlay 在诊断时可路由到甘华梁的选品专家做"新品引流"建议）
- ✅ **MiniGraph ↔ LangGraph 切换**（环境变量 `AGENT_USE_LANGGRAPH=1` 切，子图代码不动）

### 7.3 未来可扩的方向（不是承诺，是说明边际）

| 方向 | 改动范围 | 时间估 |
|---|---|---|
| 男装 / 童装 / 食品 V1 | 写一套 overlay MD + 类目基线 Wiki | 1-2 周 |
| 选品趋势聚类（抓 10w 数据做用户聚类） | 加新子图（聚类节点） + Hub 加意图分支 | 2-3 周 |
| 接入抖店 AI 助手真实对话框 | 主要是工程接入 + 推送策略，框架不动 | 跨团队协同 |
| 小二端潜力池推送 | 加一个调度任务读 evidence_refs + Review 状态机 | 1-2 周 |

## 9. 下一步计划（探索方向，非承诺）

V1 跑通了"基座 + 两条业务子图"。接下来不是堆专家、堆功能——是补**当前已经暴露出来的产品缺陷**，并探索**框架真实的扩张力**。

下面四个方向按优先级排序。**每条都标了为什么这么想 / 怎么落地 / 风险**，方便对齐分歧。

---

### 9.1 商家话术重做 · 从「指标驱动」改成「钱 + 动作驱动」

**问题在哪**

当前 V1 输出层把工程化 metric 直接抛给商家——"千川投产比 0.8 vs 基线 1.8 · 置信度 0.92"。S4/S5 老板（夫妻档亲自做）看完一脸懵：投产比是什么、基线是什么、跌 55.6% 是好是坏、置信度 0.92 又是什么意思——**这些词都是平台和小二的内部语言**。

老板想听到的只有三件事：**多少钱 · 做什么 · 几天见效**。

**商家想听到的话术 vs 当前 AI 的说法**

| 维度 | 当前 AI 的工程化说法 | 老板想听到的人话 |
|---|---|---|
| **成交** | "GMV 跌 16.4%，环比下降" | "这周比上周少卖了 16 万。按这趋势月底会差 60 万" |
| **流量** | "转化率从 2.1% 跌到 1.3%（偏离基线 -38%）" | "100 个人进店，原来能下单 2 个，现在只能下 1 个——主要是直播间没留住人" |
| **直播间** | "人均停留 22 秒 vs 类目基线 60 秒" | "观众进你直播间平均只看 22 秒就走，同行能留 1 分钟。问题不在开播时间，在福袋发得太晚" |
| **投流** | "千川投产比 0.8 / 基线 1.8 / 偏离 -55.6% · 置信度 0.92" | "你在千川每花 100 元，原来换回 160 元成交，现在只能换 80 元——继续投是赔本，今晚就停" |
| **活动** | "体验分 4.3，距离 4.5 准入门槛差 0.2" | "你差 0.2 分就能报 618 女装预热——估算多带 20w 成交。最快提分办法：下架这 3 个高退款 SKU" |
| **履约** | "退款率 8.2% > 类目均值 7.5%" | "每 100 单退 8 单，比同行高 0.7 单。退最多的是「法式碎花连衣裙」，退了 22 单——30% 是色差，要看具体原因" |
| **SKU 经济** | "SKU-001 GMV 38000 / CVR 1.1% / refund_rate 11%" | "「法式碎花连衣裙」卖了 3.8w，但退了 22 单——单算下来其实在亏钱。要么改主图色差，要么直接下架" |

**从这些例子提炼的话术原则（6 条）**

1. **用钱说话**：把比例 / 偏离改成绝对数字——"少卖了 16 万"比"GMV 跌 38%"有冲击力，比"偏离基线 -38%"易懂 10 倍
2. **拟人化对比**：把"vs 类目基线"换成"同行能…"——同行是商家心里的真实参照，"基线"是抽象数字
3. **动作必带时效 + 成本**：不是"建议优化主图"，而是"今晚就能换 / 不花钱 / 1 周入选"
4. **损益估算挂在每条建议旁**：让老板看到做了 vs 不做的差距（"估算多带 20w" / "继续投是赔本"）
5. **指标级问题要落到 SKU 级**：商家做选择题（下哪几个 SKU）比判断题（CVR 是否合格）容易得多
6. **承认局限不假装全能**："30% 是色差，要看具体原因"比"建议优化主图"更可信——商家会愿意点开看

**商家关心的指标白名单（草稿，待访谈确认）**

按"老板心智会优先在意"的四组：

| 组 | 商家会在意 | 商家**不**在意（但当前 V1 全抛出来了） |
|---|---|---|
| **赚钱** | 卖了多少 / 退了多少 / 净赚多少 / 趋势走向 | GMV / UV / AOV 缩写、环比百分比 |
| **流量** | 进店多少人 / 同行进多少 / 流量从哪来 / 哪个渠道在跌 | CVR / CTR / 跳出率、漏斗中间态绝对值 |
| **动作** | 这周做什么 / 花不花钱 / 几天见效 / 我够不够格 | 性价比 high/medium/low、置信度 0.xx、严重度 |
| **风险** | 会不会被扣分 / 哪个 SKU 在亏 / 体验分缺口 / 活动准入 | 数据完整度、data layer、evidence_refs 计数 |

**怎么落地**

1. **先列话术映射表**：10-15 条核心指标各写一份「工程话 → 老板话」对照（参考上表 7 条扩到 15 条）
2. **Composer 节点接管翻译**：把映射表 + 6 条话术原则喂给 LLM 做风格化重写，不再直接打印 raw metric
3. **input 数据全保留**：Checker 仍需要全量罗盘字段做异常检测——这是工程必要，问题只在最后输出层
4. **同一份诊断三种渲染**（一份 state，三种视图）：
   - **老板版** —— §6.1 已示例：1 句问题 + 3 条按钮 + 1 条提醒，不出现 metric 名
   - **运营版** —— 当前 Streamlit demo 形态：漏斗诊断 + 根因链 + 5-8 条建议表
   - **团队版** —— 给有专职运营的店：完整指标 + SKU 维度 + 投流明细
5. **前置 §9.4 访谈**：白名单不能我们自己拍——访谈 5-10 个真实 S4/S5 商家确认（详见 §9.4）

**风险**

- 数字越具体越容易翻车——估算"多带 20w"错了商家会骂 → 必须标"估算 / 预估 / 同行参考值"
- 不同老板话术偏好不同——有人爱听数字、有人爱听类比、有人就要表格 → 三视图设计可缓解，但需要让老板自己选默认视图
- 访谈 access 难（抖店不公开商家联系方式，走校友 / 朋友圈 / 小红书）

---

### 9.2 二人商家「消费者 + 商家」双视角选品推荐

**为什么**

S5 二人商家有一个被忽略的优势：**他们做的品类往往就是自己最熟、自己作为消费者也常买的领域**。这个"消费者直觉"是平台数据看不到的私有信号——平台知道他卖什么，但不知道他自己最爱买什么、最近收藏过什么品牌、对哪个价格带最敏感。

这个信号跟市场趋势数据**叉乘起来**，能产出"既有市场热度、又是这个老板能操盘得动"的选品建议。比传统的"纯市场趋势推荐"命中率高得多。

**和 03 PRD 已有的"个性化推荐"区别**

| 维度 | 03 PRD 的个性化 | 9.2 新方向 |
|---|---|---|
| 个性化来源 | **商家维度**（这家店历史卖过什么） | **消费者维度**（这个老板自己买什么） |
| 数据来源 | 商家 Wiki overlay | 抖音消费档案 + 老板自填问卷 |
| 适用阶段 | 在运营 | 冷启动选品 |

**怎么落地**

1. 复用甘华梁现有的商机洞察子图（不重写），加 1 个 `PersonalTasteAnalyzer` 节点
2. Hub 路由加一种意图：「个人偏好型选品」，触发条件 = 商家选择"基于我自己消费偏好选品"
3. Skill MD 加一份：`xuanpin_personal_taste.md`，定义双视角融合的 SOP
4. 数据源（按优先级）：
   - 老板填一份 5 题问卷（最近半年最常买的品类 / 最关注的品牌 / 价格带 / 最讨厌什么）—— 最稳的冷启动
   - 抖音消费档案 API（若平台开放，合规问题大）
   - 老板的关注列表 / 收藏夹（合规）

**风险**

- 抖音消费数据合规边界（个人消费数据跨域使用，PI / 隐私问题）
- 老板可能不愿填问卷（解决：把问卷做成"选品偏好测试"，把它包装成"先帮你画一张选品画像"）
- 「自己爱买的 ≠ 自己会做的」（解决：双视角融合时市场趋势仍是主权重，消费者偏好做加权）

---

### 9.3 跨子图协同 · 诊断子图触发选品子图

**为什么**

V1 的 case_5（夏树）有一个根因是"秋装备货不足，错过秋装搜索流量爆发"——当前诊断子图能识别这个问题，但**只能给"赶紧备秋装"的方向性建议**，不能给 SKU 级的"应该上哪几款秋装"。这种 SKU 级建议恰好是甘华梁选品子图的强项。

**怎么落地**

1. `DiagnosisState` 加一个 `child_subgraph_results` 字段
2. Advisor 节点遇到"备货 / 选品类"根因时（识别规则：根因 dimension ∈ {"库存", "选品", "应季"}），调用甘华梁的 `run_insight`，把 SKU 建议挂到 `actions` 里
3. Hub 加跨子图调用机制（图之间不直接通信，通过 Hub 中转）

**这是已经在 06 PRD §11 和 09 演示话术里提过的"共底座、独立编排、Hub 路由"的真正落地**——目前两个子图是平行的，没有跨调用。做完 9.3，两个子图才真正"联动起来"，框架的整合价值才被验证。

**风险**

- state 复杂度上升
- 跨子图调用的错误处理（一个挂了不能拖另一个）

---

### 9.4 商家访谈先于 schema 重构

**为什么**

PM 一贯原则：**先锁样本/用户清单，再按实际范围建技术配套**。在我们自己拍脑袋写 9.1 的"老板版指标白名单"之前，先去访谈真实 S4/S5 商家——否则我们做的是基于二手观察的设计，不是基于真实样本。

**怎么落地**

1. **样本**：通过校友 / 朋友圈 / 小红书 / 抖店商家社群，找 5-10 个抖店女装 S4/S5 店主
2. **方法**：给他们看现在的 Streamlit demo（case_1 咖喱生活），问三个核心问题：
   - 哪三个数字是你最在意的？哪些数字看完一头雾水？
   - 看到"本周建议"那段，你会真的去做哪一条？哪一条肯定不会做、为什么？
   - 如果让 AI 主动推送给你，你希望多久推一次？什么时候推？
3. **产出**：「商家关心指标白名单」+「商家不需要看的指标黑名单」+「推送频次与时机建议」
4. **回到 9.1**：用访谈结果定 owner / ops / team 三视图的具体字段

**为什么放在最后 / 优先级反而最高**

写在第 4 条只是因为它是**前置依赖**——9.1 应该等 9.4 做完再开工，9.2 也需要访谈来验证"个人消费偏好"这个假设是不是真的成立。否则做出来又是自嗨。

**风险**

- access 难、样本偏差
- 访谈结果可能颠覆 V1 的 narrative —— **这是好事**，越早被颠覆代价越低

---

### 9.5 没列入下一步的方向（说明边际）

为了避免方向蔓延，下面几个方向**确认不在 V1.x 范围**：

- **扩品类**（男装 / 食品 / 家居 V1）：技术上是 1-2 周的工作量（写一组 overlay MD + 类目基线 Wiki），但**先做"女装做扎实"比"多品类做粗糙"价值高**——9.1 + 9.4 跑完才考虑扩品类
- **跨平台**（1688 / 拼多多商家诊断）：基座可复用，但抖音平台特定的 Skill MD（自播 / 达播 / 千川）80% 需要重写——V3 量级
- **15 个专家全量上**（03 PRD §3 描述的矩阵）：V2 阶段按 ROI 分批接入，不一次性 ramp up

## 10. 成本治理 · 这事在字节内部到底能不能跑起来

任何"个性化 AI 应用"都绕不开一个问题：**token 成本会不会失控**？给字节面试官展示这套方案时，这是必须正面回答的问题。本章不讲"AI 能做什么"，只**算账 + 给压成本的具体杠杆**。

---

### 10.1 先算笔账：当前规模下 token 是多少

**单次诊断**（4 节点子图：checker → attributor → advisor → composer）

- 静态 prompt（3 个 Skill MD + 命中的 overlay + 行业 Wiki 检索片段）≈ **5-8K input**
- 动态 prompt（商家罗盘数据 + 历史 overlay）≈ **3-5K input**
- 4 节点合计 output（composer 要生成完整 markdown 报告）≈ **2-4K output**
- 合计 **10-15K token / 次**

**全量规模**（按 06 PRD §3.1：S4/S5 商家约 300w 估算）

| 诊断频次 | 调用次数 / 年 | Token 总量 / 年 |
|---|---|---|
| 每周 1 次（全量） | 1.56 亿 | 2 万亿 input + 4680 亿 output |
| 每两周 1 次（全量） | 7800 万 | 1 万亿 + 2340 亿 |
| **仅异常触发**（5% 商家/周） | 780 万 | 1000 亿 + 234 亿 |

**按 DashScope 千问 plus 外采 API 价折算**（约 4 元/百万 input + 12 元/百万 output）

| 场景 | 年成本 |
|---|---|
| 每周全量 | 约 **1.36 亿** |
| 每两周全量 | 约 6800 万 |
| 仅异常触发 | 约 **680 万** |

---

### 10.2 字节内部成本是另一个数量级

字节自家有豆包大模型——**内部调用是云原生算力成本，不是外采 API 价格**。按 GPU 时分摊估算：

- 内部成本大致是外采 API 价的 **1/3 ~ 1/5**
- 每周全量场景 1.36 亿 → 内部 **2700~4500 万 / 年**
- 异常触发场景 680 万 → 内部 **130~220 万 / 年**

对字节体量这是 **"大但不痛"**——前提是做好成本治理，否则一路涨到"小痛"甚至"刹不住"。

---

### 10.3 三个最有效的杠杆（按 ROI 排序）

#### 杠杆 A · 分层触发漏斗 —— 砍掉 95% 的 LLM 调用

不是 300w 商家每周都需要进 LLM。三级漏斗：

```
全量 300w 商家
   ↓ 规则引擎扫描异常（GMV 环比 / 千川 ROI / 体验分阈值）
异常商家 15w（5%）
   ↓ 进 LLM 诊断（4 节点子图）
高价值 / 复杂场景商家 1-2w
   ↓ 深度归因 + LLM 建议生成
```

- **第一层规则引擎**：纯 SQL + 阈值规则，零 LLM 成本。95% 商家根本没异常，没必要进 LLM
- **第二层 LLM 诊断**：只对异常商家跑 4 节点子图
- **第三层深度归因**：只对 KA 边缘 / 近期采纳率高的店做完整 root_cause_chains

**这一步把全量成本从 1.36 亿压到 680 万级别（-95%）**——是所有杠杆里 ROI 最高的一招。

#### 杠杆 B · 分层模型路由 —— 4 节点用 4 个档位模型

当前 4 节点全跑同一个模型是浪费。按"价值密度"分配：

| 节点 | 模型档位 | 为什么 |
|---|---|---|
| Checker（异常识别） | 规则引擎 + 豆包 lite / 千问 turbo | 数值比对，不需要大模型推理 |
| Attributor（归因） | 豆包 pro / 千问 plus | 需要 MECE 拆分 + overlay 理解 |
| Advisor（建议） | 豆包 pro / 千问 plus + Wiki 检索 | 主要靠 Wiki 召回，LLM 做筛选排序 |
| **Composer（话术）** | **豆包 pro / 千问 plus（不能再低）** | 商家看到的"老板话"质量崩了，整个产品价值崩塌 |

不同档位价格差 5-10 倍。**单次诊断成本能再降 40-60%**。

#### 杠杆 C · Prompt 缓存 —— 静态部分全员共享

- 3 个 Skill MD + 6 个 overlay + 4 份行业 Wiki **是所有商家共享的**，按理论可以走 prompt caching
- 千问 / 豆包 / Claude / GPT 都支持 —— 命中后 input 价格降到 **1/10**
- 我们 5-8K static + 3-5K dynamic 的结构，**缓存命中后 input 成本能降 60-70%**

技术改造成本：把 prompt 拆成「static 头 + dynamic 尾」两段，调用时按缓存 SDK 传入。**一周开发量**。

---

### 10.4 三杠杆叠加效果（推演）

|  | 调用次数 / 年 | 单次成本 | 静态部分缓存 | 年成本（内部） |
|---|---|---|---|---|
| 朴素全量 | 1.56 亿 | 100% | × | **2700-4500 万** |
| + 分层触发 | 780 万 | 100% | × | 130-220 万 |
| + 分层模型路由 | 780 万 | 40-60% | × | 60-130 万 |
| **+ Prompt 缓存** | 780 万 | 40-60% | √ | **30-80 万** |

三杠杆全部落地后：**年成本压到 30-80 万**——对字节这个体量，约等于 1-2 个工程师的人力成本。

---

### 10.5 不该砍的地方（成本边界）

省钱有边界。下面这些**砍了就毁产品价值**：

| 不能砍 | 为什么 |
|---|---|
| Composer 话术质量 | 老板看到的"少卖了 16 万"那段话是产品命脉，砍这里 = 砍产品 |
| 证据引用 evidence_refs | 砍了等于回到"AI 空话"，跟我们解的 4 大短板背道而驰 |
| Wiki 检索质量 | 规则变动 / 活动准入这些非数据信号，少一条就漏一个根因 |
| §9.1 三视图渲染 | 一份 state 三次渲染多 1-2 次 LLM 调用，但带来的分层适配价值远大于成本 |

---

### 10.6 给字节团队的成本治理建议

成本治理不是"压价"，是建立 **「成本 / 质量 / 采纳率」三维监控**：

- **每次诊断打三个指标**：token 成本（分账）+ 质量打分（小二抽检 / LLM-as-judge）+ 商家采纳率
- **看比值不看绝对值**：「成本 / 采纳条」——一次能让商家真去做的建议，花了多少钱
- 压成本压到没人采纳——**那是最贵的产品**

---

### 10.7 一句话结论

原始规模算下来 1.36 亿，听起来吓人；按 **分层触发 + 分层模型 + prompt 缓存** 三个杠杆走完，字节内部成本能压到 **30-80 万 / 年量级**。

对字节体量，这是个**"产品验证期完全可承载"**的盘子——跑通商家采纳数据后再讨论扩量；面试角度，这一节回答了"这事到底能不能跑"的硬问题。

## 12. 一句话总结

> 90% 的中小商家拿不到小二，被 AI 接管了但接管质量很烂。
>
> 本项目不是再做一个"小二副驾"——小二的 AI 排序已经够好——而是**直接补齐 AI 接管中小商家时的具体短板**：不找根因 → 加归因子图；同质化 → 类目 overlay；缺资源 → 资源数据库挂钩；不处理规则 → 双层 Wiki。
>
> 技术上是一套**可插拔的 Agent 平台**：共用基座 + Skill MD + 双层 Wiki + Hub 路由。V1 同时跑通选品（甘华梁）+ 诊断（贾丽婼）两个子图——这已经证明换品类只换 MD、换分析任务只加子图，扩展边际成本是已知的。

跑通的不是一个工具，是一种能持续扩张的产品形态。
